# 玩法承接标准

## 这是什么

玩法承接是能独立成立的、能承接题材情绪和驱动人群留存的游戏玩法系统。本文档定义玩法承接的字段、独立性判定、与机制库的关系。

参考：[00-框架定义.md](00-框架定义.md) 中的"维度 3：玩法承接"。

---

## 玩法承接 vs 品类标签

**核心区分**：

| 概念 | 数量 | 例子 |
|---|---|---|
| 品类标签 | 数十个 | "RPG"、"塔防"、"SLG"、"卡牌"、"放置" |
| 玩法承接 | 30-50 个 | "波次防守战 + 局外塔型升级"、"卡牌构筑 + Roguelike 跑图"、"资源链路自动化" |

玩法承接 = **品类标签 + 具体循环 + 商业化承接方式** 的组合。

---

## 大类分类（按品类骨架）

玩法承接按品类骨架分大类，大类代码用于构成 ID：

| 大类代码 | 品类 | 核心循环 | 典型代表 |
|---|---|---|---|
| `TD` | 塔防 | 波次防守 + 局外升级 | PVZ、向僵尸开炮、Bloons TD |
| `RPG` | 角色扮演 | 战斗 + 养成 + 探索 | 原神、明日方舟、咸鱼之王 |
| `SLG` | 策略 | 建城 + 出兵 + 联盟 | 三国志战略版、口袋奇兵 |
| `RG` | Roguelike | 单局随机 + 跨局成长 | 杀戮尖塔、吸血鬼幸存者 |
| `IDL` | 放置 | 离线收益 + 在线策略 | 一念逍遥、不朽家族 |
| `CD` | 卡牌 | 抽卡 + 构筑 + 战斗 | 阴阳师、坎公骑冠剑 |
| `SIM` | 模拟经营 | 建造 + 经营 + 自动化 | Factorio、动物餐厅 |
| `PZ` | 解谜 | 关卡推进 + 谜题设计 | 纪念碑谷、保卫向日葵 |
| `ACT` | 动作 | 操作技能 + 反馈循环 | 黑神话、艾尔登法环 |
| `BAT` | 战斗策略 | 局内即时策略 + 局外配队 | 皇室战争、英雄无敌 |

**注意**：玩法承接的本质是 "品类骨架 + 具体循环组合"，所以一个品类下可能有多个玩法承接。例如塔防 `TD` 下可以有：

- `PLY-TD-001` 波次防守 + 局外塔型升级（PVZ模式）
- `PLY-TD-002` 自走防守 + 卡组构筑（向僵尸开炮模式）

## 字段定义

每个玩法承接条目必须包含以下字段：

### 必填字段

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | 字符串 | 格式 `PLY-{大类}-{编号}`，永不修改 |
| `name` | 字符串 | 玩法承接名称（≤15字） |
| `category` | 枚举 | 大类代码（TD/RPG/SLG/RG/IDL/CD/SIM/PZ/ACT/BAT） |
| `core_loop` | 文本 | 核心循环（一句话，≤50字） |
| `independence_check` | 文本 | 独立性五问的回答 |
| `audience_ids` | 数组 | 关联的人群簇 ID 列表 |
| `theme_motif_ids` | 数组 | 适配的题材母题 ID 列表 |
| `representative_games` | 数组 | 代表游戏（≥3个） |
| `evidence_level` | 枚举 | 证据等级 S/A/B/C/D |

### 选填字段

| 字段 | 类型 | 说明 |
|---|---|---|
| `core_systems` | 数组 | 关键系统（如 `[波次防守, 局外升级, 卡牌构筑]`） |
| `monetization` | 数组 | 商业化方式（如 `[数值养成, 抽卡, 月卡, 广告变现]`） |
| `retention_depth` | 整数 1-5 | 长线留存深度（1=低，5=高） |
| `operation_burden` | 整数 1-5 | 操作负担（1=低，5=高） |
| `combinable_with` | 数组 | 可组合的其他玩法 ID（参考 [独立玩法结合法](../../../../archive/方法论/立项/独立玩法结合法_2026-05-04.md) ） |
| `mutually_exclusive` | 数组 | 互斥的其他玩法 ID（操作冲突或机制冲突） |
| `mechanism_refs` | 数组 | 关联的机制库案例路径 |
| `risk_notes` | 数组 | 主要风险（如 "操作要求高"、"内容消耗快"） |

## 独立性判定（必填字段 independence_check）

每个玩法承接必须通过 [独立玩法结合法](../../../../archive/方法论/立项/独立玩法结合法_2026-05-04.md) 中的"五问"验证：

### 五问

1. **成功案例验证**：这个玩法能否单独做成一个**成功**的游戏？
2. **操作冲突检验**：这个玩法和其他玩法结合时，是否会产生操作冲突？
3. **时间分离检验**：这个玩法和其他玩法的时间分离是否清晰？
4. **成果体现检验**：这个玩法的成果能否在其他玩法中直接体现？
5. **深度检验**：这个玩法本身是否有足够深度撑起长线？

### independence_check 字段填写格式

```yaml
independence_check: |
  1. 成功案例：✅ PVZ 系列、Bloons TD 系列证明波次防守玩法可独立成功
  2. 操作冲突：⚠️ 与操控角色战斗结合时会产生操作冲突
  3. 时间分离：✅ 波次间策略 vs 波次内自动，时间分离清晰
  4. 成果体现：✅ 局外塔型升级直接影响波次内表现
  5. 深度检验：✅ 多种塔的组合策略 + 关卡难度曲线撑起长线
```

**任何一问回答 ❌，说明这个"玩法承接"不是独立玩法，应拆分或合并。**

## 玩法承接命名原则

### 名称要求

- ≤ 15 字（比品类标签长，但比游戏名称短）
- 包含**品类骨架 + 核心循环**两部分（如"波次防守 + 局外塔型升级"）
- 避免直接用游戏名命名（如"PVZ玩法"，绑死参考游戏不可复用）

### 好的名称示例

✅ "波次防守 + 局外塔型升级"
✅ "卡牌构筑 + Roguelike 跑图"
✅ "建城出兵 + 联盟玩法"
✅ "资源链路 + 自动化优化"

### 不好的名称示例

❌ "塔防"（太宽泛，无法区分PVZ vs Clash Royale）
❌ "PVZ玩法"（绑死参考游戏）
❌ "塔防+卡牌+Roguelike"（机制堆叠，不是核心循环）

## 与机制库的关系

[机制库](../../../../archive/资料/机制库/) 是具体游戏的机制拆解，玩法承接库是抽象主题。

**对应关系**：

```text
玩法承接（抽象）   ←→   机制库（具体）
─────────────         ─────────────
PLY-TD-001            机制库/向僵尸开炮/、机制库/保卫向日葵/、...
（波次防守+局外升级）   （多个游戏案例）
```

每个玩法承接条目的 `mechanism_refs` 字段应填入对应的机制库路径：

```yaml
mechanism_refs:
  - archive/资料/机制库/向僵尸开炮/
  - archive/资料/机制库/保卫向日葵/
  - archive/资料/机制库/吸血鬼爬行者/
```

## 评分维度说明

### 长线留存深度（retention_depth）

| 分数 | 含义 |
|---|---|
| 1 | 单局体验，30分钟内体验完毕 |
| 2 | 短期内容（1-7天能消耗完） |
| 3 | 中期内容（数周内容消耗） |
| 4 | 长期内容（数月内容，但需持续运营） |
| 5 | 超长期（年级内容，赛季机制成熟） |

### 操作负担（operation_burden）

| 分数 | 含义 |
|---|---|
| 1 | 极低，纯放置或单点击 |
| 2 | 较低，标准卡牌/策略操作 |
| 3 | 中等，需要一定操作技巧 |
| 4 | 较高，需要操作熟练度 |
| 5 | 极高，硬核动作或精细操作 |

## 玩法组合规则

参考 [独立玩法结合法](../../../../archive/方法论/立项/独立玩法结合法_2026-05-04.md) 中的"结合型玩法"定义。

### combinable_with 字段填写要点

只填**已验证可组合**的玩法 ID。如果两个玩法在某游戏中组合成功，可以填入；如果只是理论上可行，标注 `(theoretical)`。

```yaml
combinable_with:
  - PLY-RG-001(theoretical)    # Roguelike + 塔防，理论可行但未见成功案例
  - PLY-CD-001                  # 卡牌构筑，已在向僵尸开炮等游戏验证
```

### mutually_exclusive 字段填写要点

明确互斥原因（操作冲突 / 机制冲突 / 节奏冲突 / 受众冲突）。

```yaml
mutually_exclusive:
  - PLY-ACT-001(操作冲突)    # 动作玩法与策略玩法操作冲突
  - PLY-SLG-001(节奏冲突)    # SLG 慢节奏与塔防快节奏不兼容
```
