# 玩法承接模板

按本模板新增玩法承接条目。完整字段定义见 [03-玩法承接标准.md](../03-玩法承接标准.md)。

---

## 模板结构

```markdown
---
id: PLY-XXX-XXX                    # 必填，分配后不可修改
name: 玩法承接名称                  # 必填，≤15字
category: XXX                       # 必填，大类代码（TD/RPG/SLG/RG/IDL/CD/SIM/PZ/ACT/BAT）
core_loop: 核心循环描述             # 必填，一句话，≤50字
independence_check: |               # 必填，独立性五问的回答
  1. 成功案例：______
  2. 操作冲突：______
  3. 时间分离：______
  4. 成果体现：______
  5. 深度检验：______
audience_ids:                       # 必填，关联人群簇 ID
  - AUD-X-XXX
  - AUD-Y-XXX
theme_motif_ids:                    # 必填，适配题材母题 ID
  - THM-XXX-XXX
  - THM-YYY-YYY
representative_games:               # 必填，代表游戏 ≥3
  - 游戏A
  - 游戏B
  - 游戏C
evidence_level: A                   # 必填，证据等级 S/A/B/C/D

core_systems:                       # 选填，关键系统
  - 系统1
  - 系统2
monetization:                       # 选填，商业化方式
  - 数值养成
  - 抽卡
  - 月卡
retention_depth: 4                  # 选填，1-5
operation_burden: 2                 # 选填，1-5
combinable_with:                    # 选填，可组合玩法 ID
  - PLY-XX-XXX
  - PLY-YY-YYY(theoretical)
mutually_exclusive:                 # 选填，互斥玩法 ID
  - PLY-ZZ-ZZZ(操作冲突)
mechanism_refs:                     # 选填，机制库案例路径
  - archive/资料/机制库/向僵尸开炮/
  - archive/资料/机制库/保卫向日葵/
risk_notes:                         # 选填
  - 风险1
  - 风险2
---

## 详细描述

### 玩法定义

（描述这个玩法承接的核心循环、玩家在其中做什么）

### 核心系统拆解

（拆解关键系统：局内、局外、循环节奏）

### 独立性论证

（详细回答独立性五问，给出具体证据）

### 代表游戏分析

（说明为什么这几个代表游戏属于这个玩法承接，它们的共同结构是什么）

### 适配的题材母题

（说明为什么这些题材母题适合这个玩法承接，情绪和系统如何对应）

### 适配的人群簇

（说明为什么这些人群簇偏好这个玩法承接）

### 组合潜力

（说明可以与哪些其他玩法组合，互斥的有哪些）

### 立项参考

（针对这个玩法承接做立项时的注意事项、典型商业化策略、风险点）
```

## 示例：PLY-TD-001 波次防守 + 局外塔型升级（参考用，不入正式库）

```markdown
---
id: PLY-TD-001
name: 波次防守 + 局外塔型升级
category: TD
core_loop: 玩家在波次间布置塔防御工事，波次内塔自动攻击，局外升级解锁新塔型
independence_check: |
  1. 成功案例：✅ PVZ系列、Bloons TD系列、保卫萝卜证明波次防守可独立成功
  2. 操作冲突：⚠️ 与操控角色战斗结合时会产生操作冲突（已在向僵尸开炮中通过"半自动+手动炮台"解决）
  3. 时间分离：✅ 波次间策略 vs 波次内自动，时间分离清晰
  4. 成果体现：✅ 局外塔型升级直接影响波次内表现
  5. 深度检验：✅ 多塔组合策略 + 关卡难度曲线 + 局外升级 撑起长线
audience_ids:
  - AUD-Y-001    # 压力释放型休闲玩家
  - AUD-Y-002    # 二次元养成抽卡用户（适配卡牌型塔防）
theme_motif_ids:
  - THM-SUR-001  # 末世求生
  - THM-POW-001  # 无双战斗
  - THM-EXP-001  # 未知探索（如保卫萝卜的探险塔防）
representative_games:
  - 向僵尸开炮
  - 保卫向日葵 (PVZ)
  - 保卫萝卜
  - Bloons TD
evidence_level: S

core_systems:
  - 局内：波次防守
  - 局外：塔型升级树
  - 局外：关卡推进
monetization:
  - 数值养成（塔型升级）
  - 月卡
  - 资源加速
  - 部分游戏有抽卡（向僵尸开炮）
retention_depth: 4
operation_burden: 2
combinable_with:
  - PLY-CD-001                  # 卡牌构筑（向僵尸开炮证明可行）
  - PLY-RG-001(theoretical)     # Roguelike + 塔防（理论可行，未见大成功）
mutually_exclusive:
  - PLY-ACT-001(操作冲突)        # 动作玩法与塔防自动节奏冲突
mechanism_refs:
  - archive/资料/机制库/向僵尸开炮/
  - archive/资料/机制库/保卫向日葵/
risk_notes:
  - 内容消耗快，需要持续推出新塔/新关卡
  - 后期数值膨胀容易导致玩家流失
  - 关卡设计成本高
---

## 详细描述

### 玩法定义

波次防守 + 局外塔型升级是一个经典的塔防玩法承接。
玩家在每个关卡开始前布置塔，波次内塔自动攻击敌人，玩家可以中途升级或新建塔。
局外通过资源积累升级塔型、解锁新塔，提升下一关卡的实力。

### 核心系统拆解

**局内系统**：
- 波次防守：每关多波敌人，波次间有短暂准备时间
- 塔布置：玩家选择塔的位置和类型
- 实时升级：波次内可以用资源升级或卖掉塔

**局外系统**：
- 塔型升级树：通过资源解锁新塔、提升现有塔
- 关卡推进：完成关卡解锁下一关
- 章节系统：每章节有不同地图和敌人

**循环节奏**：
- 单关 5-15 分钟
- 单章节 1-3 小时通关
- 全部内容数十小时（取决于游戏深度）

### 独立性论证

1. **成功案例**：✅ PVZ卖出数千万份，证明纯塔防可独立成功；保卫萝卜在国内手游也是经典案例
2. **操作冲突**：⚠️ 波次防守需要玩家在波次间快速决策，与持续操控角色战斗冲突。向僵尸开炮的解法是"自动塔 + 半自动手动炮台"
3. **时间分离**：✅ 波次间策略时间和波次内观看时间清晰分离
4. **成果体现**：✅ 局外解锁新塔后，下一关立即可以用，成就感即时
5. **深度检验**：✅ 多塔组合 + 关卡难度曲线 + 局外升级，撑起 30-100 小时游戏时长

### 代表游戏分析

向僵尸开炮、保卫向日葵、保卫萝卜、Bloons TD 的共同结构：
- 都有"布置→防守→评分→升级→下一关"的循环
- 都有"塔型多样化"的核心策略选择
- 都有"局外升级"的长线驱动力

向僵尸开炮的差异在于：加入了卡牌构筑系统，并把塔防换成了"炮台+半自动操作"，更适合手游交互。

### 适配的题材母题

- **末世求生（THM-SUR-001）**：最佳匹配。波次防守的"危机感"完美匹配末世母题的"压力情绪"
- **无双战斗（THM-POW-001）**：可适配。塔的爆发输出能给玩家"碾压感"
- **未知探索（THM-EXP-001）**：可适配，如保卫萝卜的章节探险

### 适配的人群簇

- **压力释放型休闲玩家**：偏好自动化塔防、单局短爽
- **二次元养成抽卡用户**：可适配卡牌型塔防（向僵尸开炮模式）

### 组合潜力

**可组合**：
- 卡牌构筑（PLY-CD-001）：已被向僵尸开炮验证，效果好
- Roguelike（PLY-RG-001）：理论可行，但未见大成功案例（理论可行）

**互斥**：
- 动作玩法（PLY-ACT-001）：操作冲突，难以兼容

### 立项参考

- 内容消耗快，需要持续推出新塔/新关卡
- 后期数值膨胀容易导致玩家流失，需要赛季机制平衡
- 关卡设计成本高，需要有专业关卡策划
- 商业化路径成熟：月卡 + 资源加速 + 塔型养成
```

---

## 注意事项

- independence_check 必须如实填写，任何一问回答 ❌ 说明这不是独立玩法
- 玩法名称必须包含品类骨架+核心循环，不能用游戏名命名
- audience_ids 和 theme_motif_ids 必须填写（这是三位一体的核心关联）
- evidence_level 低于 B 不能进入正式库
