# 填库工作流

## 这是什么

定义三位一体框架下，新增、修改、删除条目的标准流程。

---

## 适用范围

本工作流适用于以下三个库的所有变更：

- [人群簇库](../../../../archive/资料/人群簇库/)（按 [01-人群簇标准](01-人群簇标准.md) ）
- [题材母题库](../../../../archive/资料/买量组合库/题材库/)（按 [02-题材母题标准](02-题材母题标准.md) ）
- [玩法承接库](../../../../archive/资料/玩法承接库/)（按 [03-玩法承接标准](03-玩法承接标准.md) ）

## 新增条目流程

### 步骤 1：提案（Plan）

**主 agent 必须先 plan**：

- 说明拟新增条目的名称、所属大类、核心特征
- 说明为什么不能合并到已有条目（boundary 校验）
- 说明证据来源（代表游戏、市场观察、内部数据）
- 等待用户批准

### 步骤 2：填写草稿

按对应库的标准模板填写：

| 库 | 模板路径 |
|---|---|
| 人群簇 | [模板/人群簇模板.md](模板/人群簇模板.md) |
| 题材母题 | [模板/题材母题模板.md](模板/题材母题模板.md) |
| 玩法承接 | [模板/玩法承接模板.md](模板/玩法承接模板.md) |

**草稿位置**：先放在对应库的 `_draft/` 子目录下，文件名带 `draft-` 前缀。

### 步骤 3：自检

填完草稿后，自检以下要点：

| 检查点 | 要求 |
|---|---|
| 必填字段 | 全部非空 |
| 名称唯一 | 与现有条目无重复或近似 |
| boundary | 明确说明与相邻条目的差异 |
| 关联完整 | 引用的其他库 ID 真实存在 |
| 证据等级 | ≥ B 才能进入正式库 |
| 评分依据 | 评分有理由说明，不是拍脑袋 |

### 步骤 4：交叉引用检查

新条目如果引用其他库的 ID（如题材母题引用人群簇 ID），检查：

```text
1. 被引用的 ID 是否存在
2. 反向关系是否需要更新（例如：新增题材母题 X 引用人群簇 Y，是否需要在人群簇 Y 的"代表母题"中提到 X？）
3. 是否产生循环引用
```

### 步骤 5：分配 ID

ID 分配规则：

- 找到对应大类已有的最大编号 +1
- ID 一旦分配不可修改
- 删除条目的编号不复用

### 步骤 6：归档

- 草稿 `draft-XXX.md` 重命名为正式 ID `AUD-Y-001.md`（或对应库的 ID）
- 从 `_draft/` 移到正式库目录
- 更新 [对应库的索引](#索引文件位置)

### 步骤 7：交付

- 在最终回复中说明：新增条目 ID、关联引用更新情况、风险点

## 修改条目流程

### 允许修改的字段

参考 [04-ID与交叉引用规则.md](04-ID与交叉引用规则.md) 的"变更管理"章节。

### 修改流程

1. **提案**：说明要改什么、为什么改
2. **影响范围分析**：grep 搜索引用此 ID 的所有文件
3. **修改**：直接修改条目内容
4. **更新索引**：如果改了名称，更新索引中的名称
5. **自检**：必填字段是否仍然完整、关联是否仍然有效

### 名称修改特别注意

修改名称时，**不需要修改其他条目的引用**（因为引用用的是 ID，不是名称），但需要：

- 更新对应库的索引中该 ID 对应的名称
- 在条目顶部记录修改历史（changelog）

## 删除条目流程

### 删除前必查

```text
1. 此 ID 是否被其他条目引用（grep 全库）
2. 此 ID 是否被外部文档引用（如题材总表、评估模板）
3. 删除是否会破坏立项决策的连续性
```

### 删除步骤

如果决定删除：

1. 在引用方移除此 ID
2. 在原条目顶部添加：

```yaml
status: deleted
deleted_date: 2026-XX-XX
deleted_reason: ______
```

3. **不删除文件**，保留作废记录便于追溯
4. 在索引中标记 `[已作废]`，但保留位置

### 永远不能删除的情况

- 此 ID 已被立项决策案例引用（在 [组合案例库](../../../../archive/资料/买量组合库/组合案例库/) 中出现过）
- 此 ID 已在 [复盘库](../../../../archive/资料/复盘库/) 中出现

## 防呆机制

### 自动化校验（待开发）

后续可在 [archive/tools/scripts/](../../../../archive/tools/scripts/) 下增加自动化校验工具，校验：

```text
1. 所有引用的 ID 真实存在
2. 索引与实际文件的一致性
3. 必填字段完整性
4. 名称唯一性
```

### 手动校验（当前阶段）

在自动化工具开发前，使用 grep 命令做手动校验：

```bash
# 校验某个 ID 的引用情况
grep -r "AUD-Y-001" /Users/mt/Documents/Codex/

# 校验某个文件的引用完整性
grep -E "AUD-[A-Z]+-[0-9]{3}|THM-[A-Z]+-[0-9]{3}|PLY-[A-Z]+-[0-9]{3}" 文件路径
```

## 权限要求

| 操作 | 主 agent 权限 | 用户授权 |
|---|---|---|
| 提案（plan） | ✅ 可主动 | 不需要 |
| 填写草稿 | ✅ 可主动 | 不需要 |
| 分配 ID 和归档 | ❌ 必须用户授权 | ✅ 必须 |
| 删除条目 | ❌ 必须用户授权 | ✅ 必须 |
| 修改 reference 下的标准文档 | ❌ 必须用户授权 | ✅ 必须 |

参考根目录 [CLAUDE.md](../../../../CLAUDE.md) 的"约束先行"原则：reference 下的标准文档变更必须明确授权。
