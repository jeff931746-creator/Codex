# 长线经营假设卡

## 使用边界

- 这张卡用于在立项阶段写清长线 `ROI / 留存` 的经营模型。
- 它不是用来证明“长线结果已经成立”，而是用来回答“如果这个项目要做成长线，靠什么成立”。
- 原型阶段无法直接验证长线结果；它只能定义结构、写清验证路径，并明确止损条件。

## 什么时候使用

- 在立项定义阶段先写首版。
- 在 `01-立项评审/` 里和原型闸门一起被审核。
- 在后续小测、扩量、运营阶段持续回填和修正。

要求：

- 明确这是经营目标，不是当前已验证结论。

### 2. 玩家体验三层

> 这三层不能混写：积累什么 / 积累在哪里兑现 / 运营怎么刷新是三个独立问题，混在同一层会导致结构越拆越乱。

#### 第一层：积累（玩家持有什么）

- 局内积累的是什么：

要求：

- 必须对应具体机制，不能只写”成长感”。
- 分清”局内积累”（单局进度）和”局外积累”（跨局的持久进展）。

#### 第二层：兑现（积累在哪里发挥价值）

- 上线动因：玩家每次打开游戏的具体理由是什么：
- 下线时惦记什么：不在游戏里时，玩家心里挂着什么没解决的事：
- 付费在放大什么：玩家付钱放大的是哪个体验（不是解除哪个障碍）：

要求：

- 三个问题必须都能回溯到第一层里的某个积累机制。
- “付费在放大什么”的答案必须挂回玩法欲望，不能只写”变强”。

> 重复付费的逻辑应从"第二层：付费在放大什么"延伸，不能孤立成立。

要求：

- 付费理由必须挂回核心玩法和成长欲望。
- 不能把纯卡点和纯数值焦虑当成长线重复付费的唯一来源。

- 这里必须落到生产单元，而不是只写“持续上新”。
- 如果内容供给严重依赖重资源、重人工或高定制，必须在卡里明确写出风险。

- 哪个假设一旦失败，长线故事就不成立：

- 必须写出至少 `3` 条止损条件。

## 评审时的检查问题

- 这里写的是“经营机制”，还是“结果口号”。
- 这里写的是“可供给的内容引擎”，还是“以后再补内容”。
- 这里写的是“后续验证路径”，还是“把未来数据提前写死”。
- 这里有没有明确止损条件，还是默认项目会一直往前推。
- 这里写的"留存链"是在说明机制，还是在说时间数字（D2/D7/D30 只是结果，不是机制）。
- 第一层（积累什么）和第二层（在哪里兑现）是否被清晰区分，还是混写在一起。
- "付费在放大什么"的答案是否能回溯到第一层里的某个具体积累。
