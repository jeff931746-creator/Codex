# session-router

判断当前会话应该走哪条路。新任务先判断流程强度：`quick` 可以直接处理，`standard` / `strict` 先输出 `plan`，选定任务流程，获批后再输出后续路由决策。

## 执行步骤

### Step 0：流程强度分级

如果这是一个新任务，或当前任务方向发生明显变化：

1. 先判断任务强度：`quick` / `standard` / `strict`
2. `quick`：单步问答、查一个事实、轻量说明、无文件改动的小判断，可以直接给结论；必要时用 1 句说明假设
3. `standard`：一般分析、局部文档修改、小范围代码或脚本调整，先输出极简 `plan`，等待用户审核
4. `strict`：新项目、新目录结构、跨文件改动、部署、自动化、长期工作流规则或 Skill，必须完整 `plan`，说明目标、范围、风险、验证方式，等待用户审核
5. 用户批准 `standard` / `strict` 的 `plan` 后，再进入后续路由

如果用户明确说“直接做 / 继续 / 修掉 / 跑一下”，且任务不是高风险，可以把 `plan` 压缩为一句执行说明后继续。任务方向明显变化时，重新进入本步骤。

### Step 1：选择任务流程

`standard` 和 `strict` 任务在进入执行前，先把任务归类到一个流程类型：

- `analysis`
- `doc-change`
- `implementation`
- `review`
- `collection`

需要阶段化推进的任务如果没有完成归类，不进入执行。`quick` 任务可只做轻量状态判断。

### Step 2：检查当前门禁

`standard` 和 `strict` 任务确认记录了：

1. **任务类型**
2. **当前门禁**
3. **已完成门禁**
4. **下一门禁**

规则：

- 当前门禁未完成，不得进入下一门禁
- 如果发现想做的动作属于后续门禁，先回到当前门禁补完
- 如果任务方向变化到需要不同流程类型，重新进入 `plan`
- 正式 `plan` 和门禁切换只能由主 agent 发起与确认

### Step 3：评估当前状态

收集以下信息：

1. **任务强度**：quick / standard / strict
2. **上下文饱和度**：粗略估计当前上下文使用量（低 <40% / 中 40-60% / 高 60-80% / 满 >80%）
3. **失败计数**：同一问题连续失败几次了？
4. **任务方向**：当前做的事还在原始目标范围内吗？还是已经偏移到新目标？
5. **噪音评估**：这一步会产生大量中间输出吗？需要读很多文件但不需要修改吗？
6. **影响评估**：这一步是不是高影响、难回滚，或者已经偏离原先批准过的 `plan`？
7. **委派评估**：这一步是不是更适合子 agent，例如读 3 个以上文件、长分析、验证、校对、并行检查？

### Step 4：套用路由规则（`quick` 完成强度判断后，或 `standard` / `strict` 的 `plan` 获批且当前门禁完成后，首条命中即执行）

| 条件 | 决策 |
|---|---|
| 新任务属于 `quick` 且低风险 | **continue** |
| 同一问题连续失败 ≥ 3 次 | **rewind** |
| 任务方向根本性偏移（新目标） | **rewind** |
| 上下文 >80% 满 或 累积 ≥ 3 个失败分支 | **clear** |
| 同任务继续，上下文 >60% 满 | **compact** |
| 需要读 ≥ 3 个文件但不修改 | **subagent** |
| 任务产生大量中间输出，主上下文只需结论 | **subagent** |
| 验证 / 校对 / 文档生成 | **subagent** |
| 以上均不符合 | **continue** |

### Step 5：区分错误层级（仅当考虑 rewind 时）

- 逻辑层错误（目标方向错）→ rewind
- 实现层错误（代码 / 命令写错）→ 就地修复，不 rewind
- 理解层错误（需求没搞清楚）→ compact 后重新澄清，不 rewind

### Step 6：输出决策

格式：

```
决策：[plan / continue / rewind / compact / clear / subagent]

理由：
- 任务强度：[quick / standard / strict]
- 任务类型：[analysis / doc-change / implementation / review / collection]
- 当前门禁：[名称]
- 门禁状态：[未完成 / 已完成]
- 下一门禁：[名称]
- 上下文饱和度：[低/中/高/满]
- 失败计数：[N 次]
- 方向评估：[在轨 / 偏移]
- 噪音评估：[低 / 高]
- 影响评估：[低 / 高]
- 委派评估：[主 agent / 子 agent]

命中规则：[第几条规则]

下一步动作：
[具体说明接下来怎么做]
```

## 各决策的执行方式

**plan**：
1. 先给出简短方案，不直接开干
2. 说明预计会改哪些文件、行为或共享资产
3. 标出非显而易见的风险、权衡和回滚成本
4. 等待用户确认后再执行
5. 正式 `plan` 只能由主 agent 提交
6. `quick` 任务不强制输出正式 `plan`

**continue**：在当前已批准的 `plan` 范围内继续，不做额外切换。
只有当前门禁完成后，才能把 `continue` 用作进入下一门禁。

**rewind**：
1. 停止当前线程，不再往失败分支添加内容
2. 明确指出最后已知正确状态是哪里
3. 从该状态重新出发

**compact**：
1. 先提炼：当前目标 / 已确认约束 / 已得出结论 / 下一步
2. 运行 session-compact 写入记忆
3. 继续执行

**clear**：
1. 先运行 session-compact 保存完整状态
2. 清空上下文
3. 在新会话中从记忆摘要重建任务状态

**subagent**：
1. 明确委托内容：精确问题 + 相关文件路径
2. 规定输出格式：摘要 ≤ 300 token，或结构化 JSON
3. 指定子任务的流程类型和当前门禁
4. 只把结论带回主上下文，不带过程
5. 子 agent 不得代表整个任务提交正式 `plan`，也不得自行宣布进入下一门禁
