# 产品收集

用于把公开可检索的产品信息整理成统一结构，并写入飞书多维表格。

## 适用场景

- 收集某个平台或多个平台的产品清单
- 做竞品横向对比
- 补充上线时间、发售状态、榜单排名、收入规模
- 将结果写入飞书多维表格
- 生成可持续复用的产品调研库

## 工作流

1. 明确收集范围
- 先确认平台、时间范围、品类、是否包含未上线产品。
- 如果用户说“所有”，默认解释为“公开可检索、尽可能全量”，并明确这个边界。

2. 建立证据口径
- 优先使用官方商店页、官方公告、平台榜单、权威行业媒体。
- 能确认官方首发日就写 `官方首发日`。
- 只能确认最早公开出现时间就写 `首次公开出现日`。
- 拿不准就写 `待确认`，不要伪造精确日期。

3. 统一拆解字段
- 产品名
- 平台
- 发售状态
- 主品类
- 次级标签
- 上线时间
- 时间口径
- 地区/市场
- 数据来源
- 核心玩法
- 核心循环
- 局内体验
- 局外系统
- 主要成长线
- 内容更新方式
- 付费模式
- 主要付费点
- 社交属性
- PVP/PVE 属性
- 节奏类型
- 操作门槛
- 目标用户画像
- 核心卖点
- 参考价值
- 风险提示
- 研发复杂度
- 运营复杂度
- 榜单排名
- 收入规模
- 收入口径
- 是否适合小团队
- 题材差异化
- 买量素材强度

4. 榜单与收入规则
- 没有统一公开榜单时，不要编造具体名次。
- 可以写成：`公开销量榜待确认；商店页评测约 X 条、好评率约 Y%`。
- 收入规模只做分级判断：`头部 / 腰部 / 长尾 / 待确认`。
- 收入口径必须说明依据，例如：畅销榜区间、评测规模、活跃榜、产品形态。
- 不要伪造精确流水。

5. 表格落库规则
- 先检查飞书表是否已有相同产品。
- 新平台无法写入旧的单选字段时，优先新增兼容字段，如 `平台（统一）`。
- 已上线/未上线产品都可入表，但必须填写 `发售状态`。
- 未上线产品允许没有精确日期，`时间口径` 填 `待确认`。

## 输出要求

- 先给摘要，再给结构化结果。
- 明确哪些是官方确认，哪些是推断。
- 对“所有产品”请求，要主动说明公开检索的边界。
- 输出时优先给横向表，后给关键结论。

## 快速判断准则

### 是否适合小团队
- `是`：轻量塔防、经典关卡塔防、低系统复杂度
- `部分适合`：有一定养成或融合系统，但仍可控制范围
- `否`：强 PvP、重经营、强赛季化、多人系统复杂

### 题材差异化
- `低`：常规奇幻、传统军事、标准科幻
- `中`：题材有包装但不罕见
- `高`：中式恐怖、异世界、鲜明世界观或强风格表达

### 买量素材强度
- `低`：规则抽象、难一眼看懂
- `中`：能表达核心冲突，但不够强刺激
- `高`：题材直给、反馈强、爽点和危机感明显

## 常见注意事项

- 不要把“首次出现在榜单”的时间当成默认首发日，除非没有更好来源，并且要标出口径。
- 不要因为用户要“全量”就假装拿到了封闭平台的全部目录。
- 当平台数据口径不同，优先保留差异，不强行统一成伪精度。
- 如果要写入飞书，先确认字段是否支持该平台和状态，再录入。
