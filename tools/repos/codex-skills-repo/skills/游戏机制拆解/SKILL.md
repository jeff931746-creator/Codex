# 游戏机制拆解

对指定游戏完成机制拆解，并按机制库标准存入对应目录。

## 质量标准

所有输出必须满足 `/Users/mt/Documents/Codex/research/资料/机制库/拆解质量标准.md` 中的七条标准：

1. **机制必须可重现**：每个机制描述精确到触发条件 + 具体结果，不写感受性总结
2. **专题问题必须可否证**：用可被反驳的命题表达，不用大类名称
3. **功能分类必须有识别边界**：每个类别写清"怎么判断属于这类而不是另一类"
4. **结论必须 If-Then 格式**：可迁移到另一个游戏的设计讨论里仍然成立
5. **模块必须说清楚不负责什么**：和相邻模块的边界要明确
6. **交叉引用必须有方向和原因**：不只是标注关联，要说明依赖的是哪个具体概念
7. **模块覆盖必须和总档系统列表对齐**：总档列出的每个系统必须有对应拆解文件

## 工作流

### 第一步：确认拆解范围

在开始写任何文件之前，先确认：

- 游戏名称（用于目录命名）
- 可用的资料来源（官方资料、wiki、实际游玩、已有分析文档）
- 这个游戏最需要拆清楚的问题是什么（决定专题优先级）

如果用户提供了现有分析文档，先读完再开始，不要重复已经写好的内容。

### 第二步：写总档

用 `/Users/mt/Documents/Codex/research/资料/机制库/00_总档模板.md` 作为起点。

总档必须完成：
- 系统总览表格：每个系统写清楚"负责什么 / 不负责什么 / 和谁联动最紧密"
- 模块覆盖检查：列出每个系统对应的拆解文件计划
- 本游戏最值得拆的专题：用问句写，不用名词

总档写完后直接继续执行，不等待用户确认。完成后在最终报告里输出总档全文，供人工复核。总档是整个拆解的骨架，如果事后发现骨架有问题，需要重新跑整个流程。

### 第三步：按系统重要性决定拆解顺序

不是所有目录都要写，但必须先拆：
1. **执行层模块**（相当于小丑牌的扑克牌系统）：这是所有其他模块兑现收益的地基，最先拆
2. **路线/构筑层模块**：决定玩家"这局做什么"的核心件
3. **经济循环**：资源流转的逻辑

后期再补：
4. 关卡节奏
5. 检定系统
6. 商店与奖励
7. 结论提炼（最后写，不要一开始就写）

### 第四步：写每个专题文件

用 `/Users/mt/Documents/Codex/research/资料/机制库/专题模板.md` 作为起点。

每个专题文件必须包含：
- **这个专题要回答什么**：可否证的具体命题
- **总体判断**：1-2 句话回答主问题（这是论点，不是开头）
- **拆解正文**：每个机制有触发条件 + 具体结果 + 设计原因
- **这个模块不负责什么**：至少 2 条边界说明
- **可迁移结论**：If-Then 格式
- **相关专题**：有方向和原因的交叉引用

写完每个文件后，用质量标准的七条自检一遍再提交。

### 第五步：写结论提炼

`90_结论提炼/` 是最后写的，不是最先写的。

这里只放：
- 可以直接用于另一个游戏设计讨论的具体约束
- If-Then 格式的设计公式
- 这个游戏的结构里，去掉哪个部分整个系统会垮

不放：
- 对这个游戏的描述性总结
- 只在这个游戏语境下成立的结论

### 第六步：更新总索引

完成拆解后，更新 `/Users/mt/Documents/Codex/research/资料/机制库/00_总索引.md`：
- 在游戏索引表格里加一行
- 填写主目录路径和当前重点专题

## 两个库的关系（必读）

项目知识库（如 `向僵尸开炮知识库/`）和机制库是**两个不同层级**，不互相替代：

| | 项目知识库 | 机制库 |
|---|---|---|
| 存什么 | 这个游戏怎么做、具体设计决策、执行资料 | 这个游戏的结构拆解、可迁移结论 |
| 谁来用 | 项目组成员 | 所有游戏的设计参考 |
| 格式要求 | 项目自定义 | 必须遵循本 Skill 的质量标准 |

**规则：做了一次拆解，两个库都要写。** 项目知识库里的原始分析是输入材料，机制库里的是按标准整理后的输出结论。Jeff 做完项目知识库的拆解后，必须紧接着把结论按本 Skill 入机制库，不能只存项目目录就结束。

## 常见错误

- **先写结论再补拆解**：结论提炼必须在所有专题完成后写，不能反过来
- **把现有资料直接复制进来**：项目知识库里的原始拆解是输入，必须按质量标准重新整理后才能入机制库，不是复制粘贴
- **模块分类用名词而不是职责**：不要写"战斗系统"，要写"战斗系统负责什么、不负责什么、和哪个系统的边界在哪里"
- **结论只对这个游戏成立**："这个游戏的经济系统设计得很好"不是可迁移结论
- **只存项目目录不存机制库**：项目知识库拆解完成 ≠ 机制库入库完成，两步都要做

## 输出格式说明

每次拆解完成后，告知：
- 创建了哪些文件（路径）
- 总档的系统列表是否有未覆盖的模块
- 结论提炼里有几条 If-Then 结论
- 是否已更新总索引
