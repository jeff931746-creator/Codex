# neat-freak

在任务交付前做一次有边界的收尾检查，防止实际文件、文档规则和记忆逐渐脱节。

## 触发时机

使用本 Skill 的场景：

- 本轮改动了文档、规则、代码、脚本、Skill、目录结构或记忆。
- 重要分析、研究或拆解任务结束，并形成长期可复用结论。
- 任务产出了可交付文件、测试/评估产物或需要后续维护的工作流。

不需要触发的场景：

- quick 问答、单步事实查询、无文件改动的小判断。
- 用户明确只要临时草稿，且不会作为长期资产保留。

## 收尾步骤

1. **回看改动**
   - 用 `git status --short` 查看本轮可能涉及的文件。
   - 必要时用 `git diff` 只检查自己改动的文件。
   - 区分本轮改动和工作树里原本存在的用户改动。

2. **对齐规则和文档**
   - 如果改变了流程、目录结构、接口、脚本用法或数据口径，检查对应 `README.md`、`AGENTS.md`、`CLAUDE.md`、Skill 或 catalog 是否需要同步。
   - 如果只改了局部内容，不做全仓库整理。

3. **对齐记忆**
   - 长期有效的规则、结论、约束或任务状态应写入记忆。
   - 调试过程、失败尝试、中间输出和重复解释不要写入记忆。
   - compact 或 clear 仍使用 `session-compact`，本 Skill 不替代它。

4. **检查临时产物**
   - 留意 `.DS_Store`、缓存、临时下载、测试输出、评估报告和 `_sandbox/`。
   - 不自动删除用户未确认的文件；不确定时在交付摘要中列为待清理项。
   - 对 Skill，确认 `evals/` 和 `tests/` 不会混入 `.skill` 分发包。

5. **交付摘要**
   - 最终回复简短说明：做了什么、改了哪些关键文件、验证了什么、没动什么、还有哪些待处理项。
   - 如果有验证无法执行，说明原因。
   - **完成摘要后，必须运行检查点脚本**：
     ```bash
     bash /Users/mt/Documents/Codex/.claude/hooks/neat-freak-checkpoint.sh
     ```
   - 检查点写入后 30 分钟内有效。未写入或已过期时，`git commit` 会被 hook 阻断。

6. **经验升级判断**（Act 层）

   对照以下标准，判断本次任务是否产生了值得升级到规则层的经验：

   **升级标准（满足任意一条）：**
   - 本次纠正了一个之前反复犯的错误（同类问题第 2 次以上）
   - 发现了一个新的、会长期影响工作质量的约束
   - 本次经验与已有 feedback 记忆条目高度重叠，说明规则已该固化

   **操作：**
   - 符合升级标准 → 判断应写入哪里：
     - **只影响本工作流记忆**：更新对应 `feedback_*.md`，加强描述
     - **应成为硬规则**：直接在 `CLAUDE.md` 对应节增加或修改规则条目
   - 不符合 → 跳过，不强求每次都有升级

   > 区分："写 memory"是记录，"改 CLAUDE.md"才是真正的 Act。

## 边界

- 这是收尾质量门禁，不是全仓库清理命令。
- 不替代任务本身的测试、lint、build、smoke check。
- 不替代 `session-router` 的阶段判断，也不替代 `session-compact` 的压缩写入。
