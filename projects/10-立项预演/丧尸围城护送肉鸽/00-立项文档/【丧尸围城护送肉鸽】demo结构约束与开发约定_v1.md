# 丧尸围城护送肉鸽 Demo 结构约束与开发约定 v1

这份约束不是“玩法设想”，而是基于这个 demo 的制作历程沉淀出来的结构规则。

目标只有一个：以后继续加功能时，能保持同一条主线、同一套状态机、同一套节奏，不再把护送、商店、卡牌、镜头和战斗逻辑互相缠死。

## 1. 核心骨架不可变

- 只有一条主线路径，所有内容都必须围绕“护送推进”展开。
- 驻点、商店、检定、Boss 都是这条主线上的节奏节点，不是并行玩法。
- 战斗负责制造压力和反馈，卡牌负责制造成长分歧，路线负责制造节奏变化。
- 任何新机制都必须回答：它是在加强护送、补充肉鸽成长，还是创造驻点决策。

## 2. 状态机必须单向流转

驻点相关状态只能按下面顺序流转：

`arrival_notice -> pack_offers -> pack_opening -> pack_cards -> pack_offers`

约束如下：

- 到站先提示，再展开商店，不能直接跳到货架。
- 卡包先有开包动画，再进入抽牌界面，不能一键直达抽牌。
- 购买卡包后不自动刷新货架，刷新只能由按钮触发。
- 驻点内可连续买包，但买包和开包必须分层。
- 提示阶段不能误伤战斗；如果只是在“到站提示”，战斗仍然要继续。

## 3. 战斗与商店必须解耦

- 战斗系统只负责移动、索敌、开火、命中、击杀、受击和清场。
- 商店系统只负责货架、卡包、价格、刷新、修车和抽牌。
- 战斗可以向肉鸽系统发事件，但不能直接决定商店货架内容。
- 商店可以改变战斗数值，但不能直接篡改战斗主循环。
- 镜头、UI、商店面板、战斗画面必须彼此独立。

## 4. 数据必须先配置化，再接逻辑

所有可调内容优先落在配置层，不允许散落在主逻辑里：

- `GAME_TUNING`：全局节奏、刷怪、收益、驻点停留等基础参数。
- `jokerPool`：卡牌定义、标签、稀有度、条件、效果。
- `packCatalog`：卡包类型、费用、张数、倾向标签。
- `ROUTE` / `SPAWN_GATES`：路线与刷怪点位。

约束如下：

- 不能到处写魔法数字。
- 不能让同一个数值在多个地方以不同名字重复出现。
- 新牌、新包、新节奏参数，先进配置，再接 UI，再接战斗。

## 5. 经济循环要有明确的三段式

这个 demo 的经济循环固定为：

`击杀拿补给 -> 驻点花补给 -> 以新的压力检验 build`

具体约束：

- 到站时先提示 1 秒，再打开卡包货架。
- 驻点到达时自动刷新一次货架，之后只允许手动刷新。
- 同一驻点内买包可以不限次数，但价格必须递增，避免无限白拿。
- 卡包购买和开包结果分开结算，不能混成一个按钮。
- 买不起必须明确提示，UI 上也要同步显示缺口。

## 6. 流派必须先有体验，再有数值

流派设计不能只写“加攻击力”，必须先写清楚体验曲线。

每条流派至少要回答：

- 前期靠什么过渡。
- 中期靠什么成型。
- 后期靠什么兑现。
- 卡组里哪些牌负责底盘，哪些牌负责放大，哪些牌负责保底，哪些牌负责检定。

约束如下：

- 新卡牌必须带标签，且标签能影响商店权重、检定和成型路线。
- 每条流派都要有补洞卡，不能只有正增益没有容错。
- 有些卡必须改变节奏和交互方式，而不是只加数值。
- 流派标签至少要能从界面上被玩家一眼识别出来。

## 7. 关卡节奏必须递进

路线压力不是匀速堆高，而是分段递进。

约束如下：

- 前段给学习和成型空间，别过早把压力拉满。
- 中段必须让玩家做一次明显的取舍。
- 后段必须把压力继续推高，迫使 build 兑现。
- 驻点停留时间、检定类型、Boss 出场时机都属于节奏设计的一部分。

## 8. 镜头与 UI 必须分层

- 镜头跟随世界，UI 固定在屏幕层。
- 商店提示、开包动画、抽牌界面都是独立层，不得混入战斗画布逻辑。
- 提示文案必须和当前状态一致，不能让玩家看到“开包中”却实际上还在战斗态。
- 到站提示、开包动画、抽牌界面这三层的视觉反馈必须可区分。

## 9. 任何新功能都必须过五问

新增机制前，先回答这五个问题：

1. 它属于战斗、商店、路线、流派还是 UI。
2. 它在哪个状态被触发。
3. 它如何结束，是否会中断别的状态。
4. 它要不要进配置层，还是能直接写死。
5. 它能不能被现有的提示、检定和经济循环解释清楚。

如果这五个问题有任意一个答不清，就先别加。

## 10. 推荐的代码拆分方向

当前 demo 还是单文件实现，但内部结构应该按下面的边界思考：

- `config`：节奏、卡牌、卡包、路线、波次。
- `route`：护送路线、驻点、停留、检定。
- `combat`：移动、索敌、开火、命中、伤害、死亡。
- `economy`：补给、利息、价格、刷新、修车。
- `cards`：卡牌成长、标签、流派联动、事件加成。
- `shop`：货架、卡包、开包、抽牌、购买。
- `ui`：提示、按钮、面板、资源显示。
- `render`：镜头、世界绘制、特效、动画。

后续如果真的拆文件，优先按这个边界拆，不要按“哪段代码刚好在一起”去切。

## 11. 这套约束的底线

这套 demo 的底线不是“系统多”，而是：

- 一眼能看懂自己在哪个阶段。
- 每一次到站都能感到节奏切换。
- 每次拿牌都能感觉 build 在长大。
- 每次打怪都能感觉资源在滚动。
- 任何新内容都不会破坏“护送 + 驻点 + 肉鸽”的主骨架。

只要这五件事还成立，demo 就算继续长，也不会跑丢。
