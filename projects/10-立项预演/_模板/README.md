# 立项预演模板

新建立项预演项目时，直接复制这套按流程归档的结构。

This template follows the root workspace session protocol in [`/Users/mt/Documents/Codex/CLAUDE.md`](/Users/mt/Documents/Codex/CLAUDE.md).

这份模板的目标是让一个 demo 从立项到可试玩，始终保持同一套文件边界。

当前默认按买量型产品的早期判断逻辑来组织目录：

- 立项阶段写项目定义、验证范围和长线经营假设
- 评审阶段写原型闸门、证据缺口和决策结论
- 纸面原型阶段写服务于闸门的补证产物
- 实现阶段只写代码和配置
- 测试阶段只写验证和问题
- 发布阶段只写外链和版本

## 默认结构

1. `00-立项文档/`
   - 放正式立项文档、核心玩法定义、长线经营假设、阶段范围和风险说明
2. `01-评审/`
   - 放原型立项闸门评审、长线经营假设评审、决策记录和回滚判断
3. `02-纸面原型/`
   - 只保留两个固定子目录
   - `01-生成内容/`：放服务于闸门的补证正文、概念图 prompt、辅助脚本、参考输出和探索材料
   - `02-评审结果/`：放闸门映射、补证结论、复评建议和退回路径
4. `03-实现/`
   - 放 demo 的代码、配置、资源、模块拆分、接口说明和构建说明
   - 如果使用 Godot，优先按 `03-实现/Godot/` 里的骨架建目录
5. `04-测试/`
   - 放测试记录、Bug 清单、验证结果、调参日志和问题收敛记录
6. `05-发布/`
   - 放外链、部署说明、发布记录、版本摘要和可试玩入口

## 启动顺序

新 demo 建议先建立下面 3 类内容：

1. `00-立项文档/`
2. `01-评审/`
3. `02-纸面原型/`

只有当 `01-评审/` 最终给出“进入小测”或等价继续结论后，再补：

4. `03-实现/`
5. `04-测试/`
6. `05-发布/`

## 文件命名

建议统一使用：

`【项目名】主题_v版本号.md`

约定：

- 同一主题保留连续版本
- 不要用“最终版”“真最终版”这种模糊词
- 一个文件只回答一个主问题
- 项目名、主题名、版本号尽量稳定

## 通用边界

- `00-立项文档/` 不写实现细节
- `01-评审/` 不替代设计正文，只负责给闸门和假设下结论
- `02-纸面原型/` 不直接混入代码，也不直接宣告项目成立
- `03-实现/` 不混入太多复盘
- `04-测试/` 不写成散文式感受
- `05-发布/` 不承担设计说明

## 新 demo 检查清单

复制模板后，先确认这几件事：

- 目录边界已经建好
- 评审和正文没有混在一起
- 纸面原型内容直接服务于当前闸门
- 实现和测试的职责分开
- 发布文件只保留当前可试玩版本
