# 系统策划工作流与优化方案

本方案默认遵守根目录 [`CLAUDE.md`](/Users/mt/Documents/Codex/CLAUDE.md) 的 Session Management Protocol。

## 1. 目标

本工作流用于规范系统策划团队从需求输入到上线复盘的完整协作过程，解决以下常见问题：

- 需求来源杂乱，目标不清
- 方案写了但难以落地
- 程序、UI、美术、测试协作成本高
- 上线后只看结果，不知道问题出在哪里
- 版本迭代依赖个人经验，缺少标准化沉淀

本方案的核心原则是：

- 目标先行，而不是功能先行
- 领域负责，而不是文档负责
- 过程可追踪，而不是口头推进
- 结果可验证，而不是凭感觉复盘

## 2. 标准工作流

### 阶段一：目标输入

目标：
明确为什么做、解决什么问题、优先级是什么。

输入来源：

- 版本规划目标
- 用户反馈与流失问题
- 数据异常与指标压力
- 商业化或运营目标
- 玩法扩展与内容消耗需求

系统策划动作：

- 明确需求来源和提出方
- 识别需求属于哪个领域负责
- 判断是新增需求、问题修复还是结构优化
- 定义目标指标，如留存、活跃、付费、参与率、体验完成率
- 确认优先级、上线窗口和依赖条件

阶段产出：

- 需求卡片
- 目标定义说明
- 优先级判断
- 责任人归属

### 阶段二：问题拆解

目标：
把模糊需求转化为可分析、可设计的问题。

系统策划动作：

- 拆解用户问题、业务问题和体验问题
- 判断问题是规则问题、数值问题、入口问题、引导问题还是资源问题
- 梳理当前链路中的关键节点和断点
- 明确核心矛盾与非核心问题
- 评估影响范围和风险范围

阶段产出：

- 问题拆解文档
- 用户路径或功能路径图
- 关键矛盾清单
- 风险初判

### 阶段三：方案设计

目标：
完成业务可行、体验合理、可执行的设计方案。

系统策划动作：

- 输出系统目标和功能定位
- 设计规则、流程、边界条件和异常处理
- 设计入口组织、功能开启节奏、提示反馈和引导逻辑
- 明确与其他系统的联动关系
- 预估实现复杂度、资源量和潜在风险

阶段产出：

- 方案文档
- 流程图
- 原型图或交互草图
- 配置结构草案
- 埋点需求初稿

### 阶段四：评审对齐

目标：
让方案在开发前完成跨职能对齐，降低返工。

参会角色：

- 系统策划
- 程序
- UI
- 美术
- 数值
- 测试
- 运营或商业化相关同学

系统策划动作：

- 讲清目标、方案、边界和优先级
- 对齐实现方式与成本
- 明确哪些是必须做，哪些是可降级项
- 确认体验细节和风险点
- 形成最终版本范围

阶段产出：

- 评审纪要
- 调整后的最终方案
- 实现边界清单
- 风险与备选方案

### 阶段五：开发跟进

目标：
保证方案在实现过程中不走样。

系统策划动作：

- 跟进开发排期和节点
- 持续回答规则、边界、异常处理问题
- 跟踪 UI 和表现是否符合设计目标
- 及时处理研发中出现的妥协方案
- 维护版本中的需求变更记录

阶段产出：

- 跟进记录
- 变更记录
- 验收标准清单
- 问题修正单

### 阶段六：验收测试

目标：
确认功能可用、体验达标、数据可采。

系统策划动作：

- 对照方案进行功能验收
- 核查边界条件、异常分支和联动逻辑
- 验证入口、引导、提示反馈和表现节奏
- 检查埋点、统计口径和事件触发
- 与测试确认高风险场景覆盖

阶段产出：

- 验收清单
- 问题列表
- 埋点核查结果
- 上线建议

### 阶段七：上线验证

目标：
确认功能是否真正解决问题，而不是只完成上线。

系统策划动作：

- 观察核心指标是否达到预期
- 结合玩家反馈验证体验问题是否缓解
- 检查漏斗转化、使用路径、功能参与情况
- 识别异常数据和新的负反馈
- 形成上线初步结论

阶段产出：

- 上线观察日报或周报
- 指标对比结果
- 玩家反馈摘要
- 异常问题清单

### 阶段八：迭代优化

目标：
把上线结果转化为下一轮优化输入。

系统策划动作：

- 复盘目标达成度
- 判断问题来自规则、数值、体验、入口还是节奏
- 制定短期修复和中期优化方案
- 沉淀通用规则、模板和经验
- 反馈到后续版本规划

阶段产出：

- 复盘报告
- 优化方案
- 优先级建议
- 方法论沉淀

## 3. 优化后的工作流机制

在标准流程基础上，建议增加以下优化机制。

### 优化一：从“写文档”改为“管目标”

问题：
系统策划容易被动地变成需求执行者，交完文档后责任断掉。

优化方式：

- 每个需求必须绑定目标指标
- 每个需求必须有领域责任人
- 上线后必须由同一责任人做结果验证

优化结果：
让系统策划从“交付方案”升级为“对结果负责”。

### 优化二：增加“问题拆解”前置环节

问题：
很多团队直接从需求跳到方案，导致方案在错误问题上越做越复杂。

优化方式：

- 所有中大型需求必须先做问题拆解
- 拆清楚是业务问题、体验问题还是实现问题
- 先找关键矛盾，再做方案设计

优化结果：
减少方向性返工，提高方案有效性。

### 优化三：把基础体验并入每个领域

问题：
很多系统功能能做出来，但玩家不会用、不愿用、找不到入口。

优化方式：

- 每个方案必须包含入口设计
- 每个方案必须包含引导和提示反馈
- 每个方案必须评估认知成本和操作负担

优化结果：
避免“功能上线了但没人用”的常见问题。

### 优化四：建立评审分层机制

问题：
所有问题都在一个会上讨论，效率低，且容易漏掉细节。

优化方式：

- 小评审：策划内部确认方向和边界
- 中评审：策划、程序、UI、数值确认方案可行性
- 大评审：跨团队确认版本优先级、排期与风险

优化结果：
提升对齐效率，减少反复拉会。

### 优化五：验收从“功能对不对”升级到“体验对不对”

问题：
很多验收只看功能是否能点通，不看体验是否成立。

优化方式：

- 验收时增加体验验收项
- 覆盖入口、引导、反馈、文案、节奏、奖励感知
- 对关键功能建立体验验收清单

优化结果：
更早暴露玩家视角问题，降低上线风险。

### 优化六：上线后建立“数据 + 反馈”双验证机制

问题：
只看数据容易误判，只看玩家反馈也容易样本偏差。

优化方式：

- 每个功能上线后同时看指标和反馈
- 用漏斗数据定位问题环节
- 用反馈验证问题性质

优化结果：
提升问题归因准确度。

### 优化七：形成模板化资产

问题：
每次做需求都从头开始，沉淀效率低。

优化方式：

- 统一需求卡模板
- 统一方案模板
- 统一验收清单模板
- 统一复盘模板

优化结果：
降低沟通成本，提高团队稳定性。

## 4. 推荐分工方式

如果是中小型系统策划团队，建议按以下方式分工：

- 领域负责人：负责目标拆解、方案设计、跨部门推进、上线结果
- 执行策划：负责文档整理、配置、跟进、基础验收
- 数据支持或资深策划兼任：负责埋点、结果分析、复盘

如果是较完整的团队，建议按领域分工：

- 成长域负责人
- 社交域负责人
- 运营域负责人
- 商业化域负责人
- 战斗域负责人
- 玩法域负责人
- 数据迭代支持负责人

每个领域负责人都应遵循同一条责任链：

- 接目标
- 拆问题
- 出方案
- 推落地
- 做验收
- 看结果
- 做优化

## 5. 可直接落地的简版流程

如果你希望先快速执行，可以先用这条简化版流程：

1. 接收需求，写清目标和指标
2. 拆清问题和责任归属
3. 输出方案、流程和体验设计
4. 组织评审，确认范围和风险
5. 跟进开发，维护变更
6. 验收功能、体验和埋点
7. 上线观察数据与反馈
8. 做复盘，进入下一轮优化

## 6. 后续优化建议

这套流程后续可以继续拆成三个方向：

- 按领域细化工作流：成长域、社交域、运营域等分别细化
- 按团队职责细化模板：主策、资深策划、执行策划分别对应不同动作
- 按文档体系细化产出：需求卡、方案模板、评审模板、验收模板、复盘模板

如果需要，可以继续在这套文档基础上往下扩成完整的策划组 SOP。

## 7. 系统策划岗位约束设计

为了让系统策划团队在实际研发过程中避免职责漂移、方案失控、协作返工和结果失焦，建议为该工种建立明确的岗位约束体系。这里的约束不是限制创造力，而是确保策划工作可以稳定落地、跨团队协同、被验证和被复盘。

### 7.1 目标约束

系统策划不能只对“功能完成”负责，还必须对需求目标负责。

约束要求：

- 每个需求必须明确业务目标、体验目标或商业目标
- 每个需求必须定义可验证指标，如参与率、转化率、留存率、完成率、使用率
- 每个需求必须明确优先级和版本归属
- 每个需求上线后必须进行结果验证

执行规则：

- 没有目标定义的需求不能立项
- 没有指标定义的方案不能进入开发
- 没有复盘结论的需求不能视为闭环完成

### 7.2 职责边界约束

系统策划需要对本领域负责，但不能无限外扩，必须与玩法、数值、运营、程序等岗位建立清晰边界。

建议边界：

- 系统策划负责系统目标、规则、流程、入口、体验和跨部门推进
- 数值策划负责数值模型、公式和平衡性细调
- 玩法策划负责玩法模式、玩法循环、胜负机制和参与规则
- 运营负责活动执行、资源投放和运营排期
- 程序负责技术实现方案和实现边界判断

执行规则：

- 本领域方案由本领域责任人主导
- 涉及跨领域内容时必须与对应责任人对齐
- 不能跳过责任人直接修改他域核心设计
- 争议问题按领域归属和版本目标共同裁定

### 7.3 交付物约束

系统策划工作的核心风险之一是方案停留在口头和脑内，因此必须对交付物做统一约束。

最低交付要求：

- 需求卡
- 方案文档
- 流程图或逻辑图
- 体验说明
- 验收清单
- 复盘结论

执行规则：

- 没有标准交付物的需求不能进入评审
- 文档必须覆盖规则、边界条件、异常处理、入口、引导、提示反馈
- 口头确认不视为正式结论，关键信息必须留痕

### 7.4 流程约束

系统策划不能跳过关键阶段直接进入开发，必须遵循标准流程推进需求。

标准流程：

1. 目标输入
2. 问题拆解
3. 方案设计
4. 评审对齐
5. 开发跟进
6. 验收测试
7. 上线验证
8. 迭代优化

执行规则：

- 中大型需求必须先完成问题拆解
- 未评审通过的方案不能进入开发
- 未完成验收的内容不能进入上线节点
- 未完成复盘的需求不能定义为真正完成

### 7.5 质量约束

系统策划不只是保证功能可实现，还要保证功能可理解、可使用、可达成目标。

质量要求：

- 规则完整
- 边界完整
- 玩家认知成本可控
- 入口清晰
- 引导和反馈完整
- 数据可验证

执行规则：

- 每个方案必须包含体验检查项
- 验收必须覆盖功能正确性和体验合理性
- 没有埋点设计或无法验证效果的方案不能视为完整方案

### 7.6 协作约束

系统策划需要持续与程序、UI、美术、测试、运营协同，因此必须建立协作约束，减少返工和误解。

协作机制：

- 小评审：策划内部确认方向和边界
- 中评审：策划、程序、UI、数值确认可行性
- 大评审：跨团队确认优先级、排期和风险
- 开发期周跟进：同步变更、阻塞和降级项
- 上线后复盘会：统一问题结论和后续优化方向

执行规则：

- 需求变更必须登记
- 降级项必须同步并确认
- 风险项必须提前暴露
- 关键结论必须书面同步

### 7.7 建议优先落地的三条硬约束

如果团队目前还没有形成完整制度，建议优先执行以下三条硬约束：

- 无目标不立项
- 无评审不开发
- 无复盘不闭环

### 7.8 可直接使用的岗位约束表述

```text
系统策划岗位约束原则：
1. 对目标负责，不只对文档负责
2. 对领域负责，不越权替代他岗决策
3. 对流程负责，需求必须按标准节点推进
4. 对质量负责，方案必须覆盖规则、体验、边界和验证
5. 对结果负责，上线后必须完成数据验证与复盘闭环
```
