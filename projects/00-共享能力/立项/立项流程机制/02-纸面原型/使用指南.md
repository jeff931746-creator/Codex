## 决策树

```
从 01-立项评审 来？
    ↓
  是否给出"进入纸面原型补证"结论？
    ↓
  【是】→ 使用补证Prompt → 产出补证稿 → 使用评审Prompt → 产出评审结论
    ↓
  评审结论是什么？
    ├─ 已补足证据 → 回到 01-立项评审复评
    ├─ 同方向再补一轮 → 重新使用补证Prompt
    ├─ 核心假设被证伪 → 退回立项定义
    └─ 终止当前方向 → 项目终止
```

## 补证工作流

### 什么时候用

- `01-立项评审` 已给出 `进入纸面原型补证` 结论
- 已明确本轮要回答的闸门问题和对应阈值
- 需要用纸面原型或轻灰盒为后续复评提供直接证据

### 怎么用

1. **准备材料**
   - 项目基线（目标用户、核心承诺、核心玩法等）
   - 本轮要补的闸门（1-3 个）
   - 每个闸门的最小通过阈值
   - 上一轮评审结论

2. **使用 Prompt**
   - 复制 [补证Prompt.md](执行模板/补证Prompt.md) 的内容
   - 附上准备好的材料
   - 提交给 AI 助手

3. **产出补证稿**
   - 按照 [补证输出结构.md](输出结构/补证输出结构.md) 的格式输出
   - 保存到项目目录的 `02-纸面原型/01-生成内容/`

### 输出什么

- 纸面原型补证稿（包含项目基线、补证范围、闸门对应产物、推演记录、结论）
- 格式参考：[补证输出结构.md](输出结构/补证输出结构.md)

### 注意事项

- **不要重新发明主方向**：补证是针对性回答闸门问题，不是重新设计产品
- **不要同时保留多个主方案**：明确主方向，不要发散
- **不要改动冻结基线**：如果必须改动，需要回到 `01-立项评审` 做完整复评
- **明确本轮范围**：只回答指定闸门，明确本轮不回答的问题

## 评审工作流

### 什么时候用

- 已经有一版纸面原型补证稿
- 需要判断这版补证是否真的回答了指定闸门
- 需要给出明确结论：已补足证据 / 同方向再补一轮 / 核心假设被证伪 / 终止

### 怎么用

1. **准备材料**
   - 纸面原型补证稿
   - 原始闸门定义和通过阈值
   - 上一轮评审结论（如果有）

2. **使用 Prompt**
   - 复制 [评审Prompt.md](执行模板/评审Prompt.md) 的内容
   - 附上准备好的材料
   - 提交给 AI 助手

3. **产出评审结论**
   - 按照 [评审输出结构.md](输出结构/评审输出结构.md) 的格式输出
   - 保存到项目目录的 `02-纸面原型/02-评审记录/`

### 输出什么

- 纸面原型补证评审（包含基线一致性检查、闸门逐项评审、稳定性判断、最终建议）
- 格式参考：[评审输出结构.md](输出结构/评审输出结构.md)

### 注意事项

- **不允许用"感觉差不多"代替结论**：必须给出明确判断
- **不允许用"进入灰盒再看"代替当前判断**：纸面层面能判断的必须判断
- **不允许把结构性问题包装成数值问题**：如果是方向问题，不能用调数值解决
- **检查是否改动冻结基线**：如果改了，必须回到 `01-立项评审`

## 轻灰盒触发条件

### 纯纸面推演足够的情况

- **三秒买点**：素材脚本可以纸面推演
- **首付理由**：付费场景设计可以纸面推演
- **爽点成本**：成本结构拆解可以纸面推演

### 需要轻灰盒的情况

- **一分钟承接**：需要验证交互流畅度、操作手感
- **十分钟成型**：需要验证打法差异、成长节奏
- **二刷目标**：需要验证真实复玩动机

### 判断标准

- 如果问题是"用户能否理解" → 纸面推演足够
- 如果问题是"用户会否喜欢" → 需要轻灰盒
- 如果问题是"系统能否支撑" → 需要技术验证

详细说明：[常见问题.md](参考资料/常见问题.md#1-什么时候需要轻灰盒)

## 回退路径

### 已补足证据，回到 01 复评

**条件**：
- 本轮指定的闸门已经补足证据
- 达到了约定的通过阈值
- 未改动冻结基线

**下一步**：
- 带上补证材料回到 `01-立项评审`
- 重新评审所有六个闸门
- 重新评审长线经营假设
- 给出最终结论：进入小测 / 继续补证 / 终止

---

### 同方向再补一轮

**条件**：
- 核心假设仍成立
- 只是证据不足
- 未触发连续 2 轮失败规则

**下一步**：
- 明确本轮缺口
- 重新使用补证Prompt
- 针对性补充证据

---

### 核心假设被证伪，退回立项定义

**条件**：
- 核心假设在推演中被证伪
- 改动了冻结基线
- 需要重新设计方向

**下一步**：
- 回到立项定义阶段
- 重新定义项目基线
- 重新进入 `01-立项评审`

---

### 终止当前方向

**条件**：
- 触发一票否决项
- 连续 2 轮补证仍未通过
- 多个核心闸门明确不成立

**下一步**：
- 项目终止
- 总结失败原因
- 归档材料

## 常见场景

### 场景 1：首次补证

1. 从 `01-立项评审` 获得 `进入纸面原型补证` 结论
2. 明确本轮要回答的闸门（1-3 个）
3. 使用 [补证Prompt](执行模板/补证Prompt.md)
4. 产出补证稿
5. 使用 [评审Prompt](执行模板/评审Prompt.md)
6. 根据评审结论决定下一步

---

### 场景 2：迭代补证

1. 上一轮评审给出 `同方向再补一轮` 结论
2. 分析上一轮缺口
3. 使用 [补证Prompt](执行模板/补证Prompt.md)，附上：
   - 上一版补证结论
   - 本轮改动点
   - 为什么要改
4. 产出新版补证稿
5. 使用 [评审Prompt](执行模板/评审Prompt.md)
6. 根据评审结论决定下一步

---

### 场景 3：需要轻灰盒

1. 纸面推演发现某个闸门需要轻灰盒验证
2. 在补证稿中明确：
   - 哪个闸门需要轻灰盒
   - 轻灰盒的最小范围
   - 验证目标
3. 制作轻灰盒
4. 验证后更新补证稿
5. 使用 [评审Prompt](执行模板/评审Prompt.md)

---

### 场景 4：改动了冻结基线

1. 补证过程中发现必须改动冻结基线
2. 在补证稿中明确标注：
   - 改动了哪些基线
   - 为什么必须改动
3. 评审时明确指出：必须回到 `01-立项评审` 做完整复评
4. 不能当作普通补证评审
