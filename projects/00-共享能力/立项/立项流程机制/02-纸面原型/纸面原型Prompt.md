# 纸面原型Prompt

该 Prompt 对应 `立项流程机制` 的第 `02` 步：`纸面原型`。

## 适用场景

- 已完成立项前置评审，且 `方向卡 / 母型卡 / 参与方式卡 / 核心机制卡 / 玩法循环卡` 已有明确结论
- 需要在进入白模、灰盒或可玩原型前，先做一轮纸面原型收敛
- 已有 `纸面原型 v1 / v2`，需要判断本轮更新到底是优化、结构调整，还是已经推翻上一版
- 目标不是继续发散创意，而是把评审结论压缩成一版最小、可推演、可否定的单局模型

## 使用建议

- 推荐先附上评审稿或立项文档中的核心结论，再使用下面的 Prompt
- 如果材料不完整，也可以先跑一版，但必须允许模型把缺失信息标为 `待确认`
- 如果是迭代版，请同时附上 `上一版纸面原型结论 + 本轮改动点 + 触发改动的问题`
- 如果输出要落到 `立项预演` 项目目录，默认保存到 `02-纸面原型/01-生成内容/`

```md

## 硬约束

- 高频动作不超过 `3` 个
- 关键决策不超过 `2` 个
- 核心机制必须在前 `3-5 分钟` 内可感知
- 差异点必须进入主循环，而不是停留在包装层
- 循环节点不超过 `5` 个
- 单局里必须交代：目标感、第一次爽点、差异机制出现时间、最终收束方式、下一轮动机
- 如果删掉核心机制后项目变化不明显，必须在结论里指出这个问题
- 如果纸面原型已经显示主循环不成立，不要硬写成“可继续”
- 如果我提供了上一版方案，必须显式比较 `上一版结论 -> 本轮变化 -> 新结论`
- 每个关键改动必须归类为：`局部优化 / 结构调整 / 结论推翻`
- 当改动影响 `玩家身份 / 主母型 / 核心机制 / 主循环节点 / 单局胜负目标 / 核心爽点来源` 时，不能当作普通优化，必须触发重评
- 必须定义：什么情况下继续当前版本，什么情况下回滚上一版，什么情况下退回立项评审的上一阶段

## 输出风格

- 用内部评审文档语气，简洁、直接、能拿去开会
- 少写愿景表达，多写验证逻辑
- 优先写“本轮只验证什么、不验证什么”
- 判断和建议要能落到下一步动作
- 如果发现本轮更新已经把项目变成另一套东西，要直接指出，不要用“迭代”轻描淡写带过
```
