# 通用 Demo 文件规范与边界条件 v1

这是一份面向后续所有 demo 制作的通用模板，不绑定任何单一项目。

它的用途不是描述某个玩法，而是规定：

- 文档应该怎么分层
- 每类文件应该写什么
- 哪些边界不能越过
- 新 demo 开始时，先建哪些文件

## 1. 适用范围

这份规范适用于所有 demo、纸面原型、玩法验证、战斗原型、数值验证、UI 原型和可试玩样板。

适用对象包括：

- 新立项的玩法验证 demo
- 已有项目中的独立子 demo
- 需要反复迭代的交互原型
- 需要交付给生成式工具、程序员或策划共同协作的样板

不适用对象：

- 纯文案稿
- 纯视觉 moodboard
- 单纯的市场宣传材料
- 与玩法无关的通用知识笔记

## 2. 标准目录边界

新 demo 建议采用下面的目录骨架：

```text
项目名/
  00-立项文档/
  01-评审/
  02-纸面原型/
  03-实现/
  04-测试/
  05-发布/
```

目录边界约定：

- `00-立项文档/` 只放目标、方向、机制、范围、风险和版本结论。
- `01-评审/` 只放评审稿、评审意见、结论、回滚记录。
- `02-纸面原型/` 只放视觉草图、流程图、文案稿、生成 prompt 和参考图。
- `03-实现/` 只放代码、配置、资源、构建说明、模块拆分和实现说明。
- `04-测试/` 只放测试记录、Bug 清单、验证结果、调参日志。
- `05-发布/` 只放外链、部署说明、发布记录、版本摘要。

如果 demo 规模很小，可以不建满全部目录，但不要把不同性质的内容混在一起。

## 3. 文件命名规范

所有 demo 文件统一采用：

`【项目名】主题_v版本号.md`

约定如下：

- 文件名中的 `项目名` 必须稳定，避免同一项目出现多个别名。
- `主题` 只写一件事，不要把多个目标塞进同一个标题里。
- 版本号统一使用 `v1 / v2 / v3`，不要混用 `最终版 / 新版 / 修正版 / 真的最终版` 这类模糊命名。
- 同类文件必须可排序、可回溯、可比较。

建议：

- 文档正文用中文命名
- 目录名尽量固定
- 不要在文件名里塞过长的解释句

## 4. 每类文件应该回答什么

### 4.1 立项文档

立项文档只回答四个问题：

- 这个 demo 想解决什么问题
- 它的核心体验是什么
- 这次验证的边界在哪里
- 什么情况下要停，什么情况下要继续

立项文档不应该写：

- 具体代码实现
- 详细 UI 细节
- 大量数值表
- 不相关的历史复盘

### 4.2 评审文件

评审文件只回答：

- 是否值得继续
- 哪些风险不能接受
- 哪些内容需要回收
- 下一轮优先改什么

评审文件不应该替代立项文档，也不应该当作设计正文。

### 4.3 纸面原型

纸面原型只回答：

- 玩家第一眼看到什么
- 核心交互长什么样
- 16:9 或 9:16 的画面里各层怎么摆
- 如果给生成式工具，它要按什么约束出图

纸面原型不应该直接变成实现代码，但必须能被转成实现输入。

### 4.4 实现文件

实现文件只回答：

- 代码如何组织
- 模块怎么拆
- 哪些参数可调
- 哪些行为由配置驱动

实现文件不应该混入过多策划复盘。

### 4.5 测试文件

测试文件只回答：

- 这版改动验证了什么
- 哪些问题已解决
- 哪些问题还存在
- 下一次测试重点是什么

测试文件必须尽量客观，少写感受，多写现象和结果。

### 4.6 发布文件

发布文件只回答：

- 版本号是什么
- 外链在哪里
- 部署到了哪里
- 当前是否可试玩

发布文件不应该替代设计文档。

## 5. 文档边界条件

### 5.1 一份文件只回答一个主问题

如果一个文件同时在回答“玩法设计”“UI 原型”“实现结构”“发布说明”，就说明拆分不够。

### 5.2 一个主题只保留一个主版本

同一主题下，保留最新的结构化版本。

旧版本不要删除，除非已经确认完全失效。

### 5.3 先正文，后附录

正文只放结论和规则。

附录可放：

- 参考链接
- 旧方案对比
- 备选方案
- 额外说明

### 5.4 先通用规则，后项目特化

通用模板里先写所有 demo 都要遵守的规则，再在项目附录里写本项目特有的例外。

## 6. 资源与文件边界

### 6.1 允许放入 workspace 的内容

- 文档
- 图片
- 结构化表格
- 配置文件
- 代码
- 脚本
- 必要的参考材料

### 6.2 不建议放入 workspace 的内容

- 安装包
- 临时运行时
- 大型缓存
- 下载后未整理的二进制堆积
- 与 demo 无关的个人杂项

### 6.3 临时文件的边界

临时内容统一放在 `tmp/`，用完尽量清掉。

如果临时文件变成长期材料，就迁回正式目录，不要继续堆在临时区。

## 7. 结构约束

### 7.1 先定边界，再写内容

新 demo 开始时，先确定：

- 要不要分 `立项 / 评审 / 原型 / 实现 / 测试 / 发布`
- 哪些目录一定要建
- 哪些目录可选
- 哪些文件属于正文，哪些属于附录

### 7.2 先定版本，再扩内容

每一轮迭代都必须明确版本号。

不要把多个版本揉在一个文件里，也不要把历史记录直接覆盖掉。

### 7.3 先定输入，再定输出

如果文件是给生成式工具、程序员或策划用的，必须先写：

- 输入是什么
- 输出格式是什么
- 允许偏离的范围是什么

## 8. 新 demo 启动清单

每个新 demo 建议先建这 4 类文件：

1. 立项文档
2. 纸面原型
3. 结构约束与文件规范
4. 评审记录占位文件

如果是可试玩 demo，再补：

5. 实现说明
6. 测试记录
7. 发布说明

## 9. 边界禁止项

以下情况应尽量避免：

- 一个文件同时承担设计、实现、测试、发布四种职责
- 版本号混乱，导致无法判断最新稿
- 文件名不断变体，但内容没真正推进
- 在正文里塞太多无关例子，导致规范变得像散文
- 把项目复盘写成了以后所有 demo 都能套用的规则
- 把通用模板写成某个项目的玩法说明

## 10. 推荐使用方式

新 demo 开始时，建议按这个顺序做：

1. 先建立目录骨架。
2. 再写立项文档。
3. 再写纸面原型。
4. 再写结构约束。
5. 再进入实现。
6. 每次大迭代都补测试和发布记录。

如果后续 demo 已经在跑，先补目录边界和文件命名，再补正文。

## 11. 可直接复用的判断标准

看到一份 demo 相关文件时，先问自己：

- 它属于哪个目录
- 它只回答了一个问题吗
- 它有没有明确版本号
- 它是不是把通用规则和项目特化混在一起了
- 它是否适合被未来另一个 demo 直接复用

如果这五个问题里有两个以上答不清，这份文件就还不够规范。
