# Steam 卖相与 Hook 立项方法论

来源视频：[在Steam上什么游戏卖得好：你并不需要一个“钩子”](https://m.bilibili.com/video/BV1kMyzBxEJN)  
UP：游戏开发-  
沉淀日期：2026-04-30  

## 一句话结论

独立游戏立项不应该从“我有一个独特玩法钩子”开始，而应该从“玩家一眼能理解并愿意购买的体验承诺”开始。

```text
玩法钩子 ≠ 卖点
体验承诺 = 卖点
Hook = 放大器，不是需求本身
```

## 核心定义

### Hook

这里的 Hook 指一个能让游戏显得不同的机制点，例如特殊操作、新奇规则、反常识组合，或能在 trailer 里快速展示的机制。

Hook 能回答：

```text
这游戏哪里不一样？
```

但 Hook 不一定能回答：

```text
我为什么想玩？
我为什么现在要买？
```

### 体验承诺

体验承诺是玩家真正购买的对象。它通常对应一种身份、幻想、情境、成长或情绪。

更好的立项表达不是：

```text
这是一款有 X 创新机制的游戏。
```

而是：

```text
这是一款让你体验 X 幻想的游戏，机制服务于这个幻想。
```

## 为什么只追求 Hook 很危险

Hook 容易制造假确定性：

```text
我有一个别人没做过的机制
所以它有市场
```

但市场判断真正要问的是：

```text
有没有人想要？
玩家是否能立刻理解？
截图和短视频能不能传达？
这个体验够不够大？
做出来的成本是否可控？
```

常见失败形态：

- 只有机制，没有可购买的幻想。
- 只有新奇，没有持续乐趣。
- 只有复杂设定，没有清晰受众。
- 只有差异化，没有 Steam 页面吸引力。
- Scope 太大，团队做不到足够完成度。

## 四大支柱

| 支柱 | 作用 | 失败表现 |
|---|---|---|
| Fun | 让玩家玩下去 | 试玩后无聊，留存差 |
| Appeal | 让玩家点进来、愿望单、购买 | 页面没人点，trailer 没吸引力 |
| Scope | 让团队做得完、做得好 | 内容做不完，质量被摊薄 |
| Monetization | 让产品能卖钱 | 定价、受众、平台不匹配 |

关键判断：很多开发者过度关注 `Fun`，但 Steam 上第一道门槛是 `Appeal`。玩家还没玩到游戏前，只能通过标题、封面、截图、标签、trailer、简介、题材和视觉风格判断是否值得点进来。

如果 `Appeal` 不成立，`Fun` 再好也很难被看见。

## 推荐立项顺序

错误顺序：

```text
想到一个机制
→ 做 demo
→ 再想怎么包装
→ 上 Steam 后发现没人点
```

更稳的顺序：

```text
市场/受众需求
→ 体验承诺
→ 可视化表达
→ Scope 判断
→ 核心机制
→ 内容生产
```

## 五种可用立项路径

### 1. 成熟类型迭代

在已有需求上做更清晰、更小、更漂亮、更聚焦的版本。

适合问：

```text
这个类型的玩家已经在买什么？
他们抱怨什么？
我们能不能用更小 scope 解决一个明确痛点？
```

优点：需求已验证。  
风险：竞争强，需要执行质量。

### 2. 强幻想切入

先找到玩家想体验的身份或情境。

例：

```text
我想建造并守住自己的城堡
我想经营怪物医院
我想操控一群越来越离谱的单位
我想成为末日列车指挥官
```

优点：容易做 Steam 页面和 trailer。  
风险：幻想大但系统支撑不住，会变成空壳。

### 3. 找市场空位

不是追求完全创新，而是在已有需求里找缺口。

例：

```text
玩家想要 A 类型，但不想要它太复杂
玩家想要 B 题材，但缺少轻量版本
玩家喜欢 C 机制，但缺少某种美术/叙事包装
```

优点：更可能形成商业机会。  
风险：空位可能是因为没人想要，不是没人发现。

### 4. 先做 Appeal

先验证一句话、截图、trailer、Steam capsule 是否吸引人。

判断逻辑：

```text
如果看起来都不想玩，后续完整开发只是在放大风险。
```

### 5. 技术优势转体验优势

技术本身不是卖点，技术带来的玩家体验才是卖点。

错误表达：

```text
我们有复杂的物理模拟。
```

更有效表达：

```text
你可以摧毁、燃烧、触发连锁反应，像真的掌控一个危险世界。
```

## 立项检查表

每个新点子先过这 10 问：

1. 玩家幻想是什么？
2. 一句话能不能讲清？
3. 一张图能不能看懂？
4. Steam 标签能不能落到已有需求？
5. 竞品是否证明有人买？
6. 我们和竞品的差异在哪里？
7. 差异是体验差异，还是只有机制差异？
8. 最小可售版本需要多少内容？
9. 这个 scope 是否符合团队能力？
10. Hook 是否服务体验，而不是替代体验？

## 对当前立项流程的影响

后续评估项目时，不先问：

```text
这个游戏的钩子是什么？
```

先问：

```text
玩家第一眼想买的体验是什么？
这个体验是否能被截图、标题和 15 秒视频表达？
这个体验背后是否有已验证需求？
我们能不能用可控 scope 做出足够完整的版本？
```

如果这些问题回答不了，问题通常不是“缺一个钩子”，而是 `Appeal` 还没成立。

## 使用方式

- 立项早期：用“立项检查表”筛掉只有机制、没有体验承诺的点子。
- Steam 页面预演：先验证一句话、封面、截图和 trailer 是否能表达体验。
- Scope 评审：在机制兴奋前先估最小可售版本内容量。
- 竞品拆解：拆解竞品时优先看它卖的体验，而不是只看系统结构。
