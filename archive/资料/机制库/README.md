# 机制库说明

该库用于做**按单个游戏展开的专题性结构拆解**，目标不是堆资料，也不是按游戏类型归类，而是把每个游戏的核心结构、模块分工、流派形成、经济循环、关卡节奏、检定方式拆清楚。

它更接近一份“可复用的游戏拆解模板库”。

## 总原则

- 一个游戏对应一个主目录
- 一个游戏必须先有总档，再有专题拆解
- 专题只围绕一个问题，不允许一篇同时回答多个大问题
- 专题必须能回挂到具体系统模块，不能悬空
- 模块、流派、经济、关卡、检定必须分开写，不能混在一篇里
- 参数和变体只作为补充，不作为主结构

## 推荐结构

```text
机制库/
  00_总索引.md
  游戏名A/
    00_总档.md
    01_系统结构/
    02_模块拆解/
    03_流派设计/
    04_经济循环/
    05_关卡节奏/
    06_商店与奖励/
    07_Boss与检定/
    90_结论提炼/
    99_归档/
  游戏名B/
    ...
  90_专题串联/
  99_归档/
```

现有的 `肉鸽/` 可以先视为历史专题资料区，新的游戏拆解请优先按 `小丑牌/` 这种主目录样式新开目录。

## 目录职责

### `00_总索引.md`

总索引只做两件事：

- 告诉你这个库里有哪些游戏
- 告诉你每个游戏分别有哪些专题

总索引不写分析正文。

### `游戏名A/00_总档.md`

每个游戏必须先有总档。

总档只回答游戏本身的骨架问题：

- 这是什么游戏
- 它靠什么成立
- 核心循环是什么
- 关键系统有哪些
- 这些系统如何互相咬合
- 本游戏最值得拆的专题是什么

### `01_系统结构/`

放“这个游戏有哪些大系统”的拆解。

例如《小丑牌》可拆为：

- 扑克牌系统
- Joker 系统
- 星球牌系统
- 塔罗牌系统
- 幻灵牌系统
- 商店系统
- Boss Blind 系统
- 经济系统
- 关卡/盲注结构

### `02_模块拆解/`

放单个系统模块的拆解。

每个模块只讲一件事：

- 它负责什么
- 它怎么运作
- 它和谁联动
- 它的边界是什么

### `03_流派设计/`

放模块如何组合成流派。

必须写清：

- 流派入口
- 流派核心件
- 流派放大件
- 流派稳定件
- 流派短板
- 流派如何被检定

### `04_经济循环/`

放资源怎么来、怎么花、怎么滚雪球、怎么反过来影响构筑。

至少要回答：

- 收入来源是什么
- 支出入口是什么
- 购买时机是什么
- 经济如何影响成型速度
- 经济如何影响节奏

### `05_关卡节奏/`

放局内推进节奏、阶段切换、压力爬升和关卡结构。

需要拆清：

- 前期、中期、后期分别干什么
- 压力怎么上升
- 什么时候给喘息
- 什么时候逼转型

### `06_商店与奖励/`

放奖励节点、修正节点、补强节点。

这里重点看：

- 奖励如何服务流派
- 商店如何修正路线
- 奖励是否能补结构短板

### `07_Boss与检定/`

放专门打偏科、检验鲁棒性的内容。

需要说明：

- 它检定什么
- 它克制什么
- 它怎样逼玩家改变思路

### `90_结论提炼/`

放可迁移结论、设计公式、复用模板。

这一层是给后续项目直接拿来用的，不再是原始拆解。

### `99_归档/`

放旧版本、重复稿、失效分析，保持主干清爽。

## 写作约束

**内容质量标准见 → [拆解质量标准.md](拆解质量标准.md)**

以下是结构层面的约束，内容层面的合格标准（机制可重现、问题可否证、结论可迁移等）在质量标准文档里。

### 1. 一个专题只回答一个主问题

例如：

- `小丑牌_牌种体系专题`
- `小丑牌_流派形成专题`
- `小丑牌_经济循环专题`
- `小丑牌_关卡与盲注专题`

不要把“系统、流派、经济、关卡”混在一篇里。

### 2. 专题必须回挂到总档

每篇专题必须能回到：

- 总结构
- 相关系统模块
- 相关流派或节奏节点

不能只写结论，不写来源位置。

### 3. 不把参数当机制本体

像下面这些都不是主结构：

- `三选一 / 四选一 / 五选一`
- `高概率 / 低概率`
- `一次性 / 多次`
- `刷新 / 不刷新`

这些只能作为参数说明，不能单独撑起一个专题。

### 4. 不把品类当主分类

品类只能做辅助标签，不能做一级目录逻辑。

因为同一个游戏的拆解重点，往往不在品类名上，而在：

- 系统怎么分工
- 流派怎么长出来
- 经济怎么滚
- 关卡怎么压

### 5. 先总档，后专题，最后再提炼结论

顺序必须固定：

1. 先写总档
2. 再拆系统
3. 再拆专题
4. 再写结论提炼

不要反过来。

### 6. 专题之间允许交叉引用，但只能有一个主文件

比如一个模块既影响流派，也影响经济：

- 只能有一个主文件
- 其他专题里引用它
- 不允许复制一份重复内容

### 7. 目录名优先按游戏名和专题名命名

推荐命名格式：

- `游戏名/00_总档.md`
- `游戏名/03_流派设计/`
- `游戏名/04_经济循环/`

专题名尽量直接表达问题，不要抽象成学科名。

### 8. 模块覆盖必须和总档系统列表对齐

总档"主要系统"列出的每个系统，必须在 `02_模块拆解/` 里有对应文件。

没有对应文件 = 该游戏拆解有盲区，必须补写或在总档里注明"暂未拆解"。

### 9. 系统总览必须同时写清楚每个系统的边界

总档的系统总览不只是列名称，每条必须包含：

- 它负责什么
- 它不负责什么（和哪个相邻系统的边界在哪里）
- 和谁联动最紧密

一句话写清楚两个方向（是什么 + 不是什么），才算有效的系统总览。

## 《小丑牌》示例

如果按这套结构，《小丑牌》应该这样拆：

- `00_总档`
- `01_系统结构`：4 种牌体系、商店、Boss Blind、经济、关卡结构
- `02_模块拆解`：扑克牌、Joker、星球牌、塔罗牌、幻灵牌
- `03_流派设计`：不同牌种如何组合成流派
- `04_经济循环`：金币、刷新、利息、购买、提速
- `05_关卡节奏`：盲注推进、阶段压力、Boss 检定
- `06_商店与奖励`：商店修正和奖励节点如何改变路线
- `07_Boss与检定`：如何专门打偏科 build
- `90_结论提炼`：可迁移到别的游戏的骨架结论

## 使用建议

- 每新增一个游戏，先建主目录和总档
- 再决定这个游戏需要哪些专题
- 专题数量不固定，但必须足够解释这个游戏的关键结构
- 如果一个专题写不满，就说明它不该单独成篇
