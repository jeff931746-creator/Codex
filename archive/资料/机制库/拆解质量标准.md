# 拆解质量标准

这份文档规定了机制库里每篇专题**写到什么程度才算合格**。它不是关于怎么组织文件，而是关于怎么判断内容有没有真正拆清楚。

---

## 核心判断原则

> **一篇拆解的价值，不在于它说了多少，而在于不玩这个游戏的人能否凭它重现这个机制的行为，以及做另一个游戏的设计者能否从中提取可操作的约束。**

---

## 七条质量标准

### 标准一：机制必须写到可重现

每个机制描述都应该精确到可以被另一个人实现或验证。

**不合格示例**：
> 利息机制让省钱变成主动策略。

**合格示例**：
> 每轮结束时，持有金币每满 $5 获得 $1 利息，最多 $5/轮。持有 $25 时利息上限触顶。

**检验方法**：把这段描述给一个没玩过这个游戏的人，他能说出"在什么条件下触发、产生什么具体结果"吗？不能就没写清楚。

**空位词警报**：以下词汇出现时，说明机制描述还没写到位，必须替换成具体内容：

- "特定植物 / 特定等级 / 特定条件" → 是哪个植物？哪个等级？什么条件？
- "某些技能 / 一定程度 / 较高倍率" → 具体是哪些？多少？
- "满足要求后 / 达到阈值时" → 要求是什么？阈值是多少？

有空位词 = 机制还没写清楚，只写了壳子没写内容。

---

### 标准二：专题问题必须可被否证

文件开头的"这个专题要回答什么"，必须是一个可以被反驳的具体命题，而不是一个大类名称。

**不合格示例**：
> 这个专题要回答 Joker 的设计。

**合格示例**：
> 这个专题要回答：Joker 为什么是流派发动机，以及 5 槽约束如何把"变强"从堆叠变成取舍。

**检验方法**：这个问题能被回答"不是"或"不对"吗？能，就是合格的具体命题。不能，就还是一个大标题。

---

### 标准三：功能分类必须有识别边界

每当对某个系统内的元素做分类（如 Joker 功能分类、塔罗牌效果分类），每个类别都必须说清楚**怎么判断一张牌属于这类而不属于另一类**。

**不合格示例**：
> 路线件：负责声明路线方向。

**合格示例**：
> 路线件的识别标准：（1）拿到后，后续商店购买决策会发生改变；（2）它对一类玩法有强加成，同时使另一类玩法变成浪费；（3）能和至少两个其他系统联动。

**检验方法**：拿一个具体例子，能用这个标准判断它属于哪类、不属于哪类吗？如果分类标准不能排除任何东西，它就没有信息量。

---

### 标准四：结论必须用 If-Then 格式写，且 If 部分必须包含游戏专属的机制条件

`90_结论提炼/` 里的结论，以及专题末尾的"总体判断"，必须可以被直接拿到另一个游戏的设计讨论里使用。

结论有两个方向的失败，都不合格：

**失败方向一：太窄——只在这个游戏里成立**

> Boss Blind 是《小丑牌》里最关键的偏科检定器。

**失败方向二：太宽——对任何游戏都成立，没有提炼出这类游戏的独特机制洞察**

> 如果合成系统会改变后续资源投放方向，它就已经是路线层。

这条对任何有合成系统的游戏都成立，没有说清楚"什么样的合成系统会产生这个效果、什么样的不会"。

**合格示例**：
> 如果一个游戏把合成操作设计成局内的一次性路线声明——选了这条合成路线就使另一条合成路线的收益下降——则这个合成系统就不只是内容解锁，而是构筑决策层。判断标准：合成结果是否改变了后续的最优选择集合？改变了，是路线层；没改变，只是收集层。

**检验方法（双向）**：
- **太窄测试**：把这句结论放到另一个完全不同的游戏设计讨论里，它还成立吗？不成立说明结论还没抽象到位。
- **太宽测试**：把 If 部分里的游戏类型换成完全不同的游戏类型，结论是否仍然100%成立且不提供额外判断依据？如果是，说明 If 条件还没有精确到这类游戏的机制特征。

---

### 标准五：模块必须说清楚它不负责什么

每个模块拆解，除了说"它负责什么"，还要说"它不负责什么"以及"和它最近的模块的边界在哪里"。

**不合格示例**：
> 星球牌负责提升底盘，给 build 提供稳定性。

**合格示例**：
> 星球牌负责提升特定牌型的基础筹码和倍率，但不改变单张扑克牌的属性（那是塔罗牌的职责），也不声明路线方向（那是 Joker 的职责）。它只做一件事：让选定路线的天花板足够高，使 Joker 的倍率有意义地兑现。

**检验方法**：如果去掉"不负责"的部分，还能区分这个模块和相邻模块吗？不能，就还没写清楚边界。

---

### 标准六：交叉引用必须有方向和原因，且描述必须准确指向目标文件

文件末尾的"相关专题"不只是标注关联，还要说明为什么相关、依赖的是哪个具体概念。括号里的说明必须是关于**目标文件内容**的准确描述，不能写成关于当前文件或其他文件的内容。

**不合格示例一（缺少原因）**：
> 相关专题：→ 经济循环拆解

**不合格示例二（原因写错了文件）**：
> → [植物成长底盘拆解](植物成长底盘拆解.md)（杂交重写的是执行层单位本身）

括号里说的是执行层，不是成长底盘。

**合格示例**：
> → [经济循环拆解](../04_经济循环/经济循环拆解.md)（商店里的每次花钱决策依赖经济循环里的利息机制来权衡时机）

**检验方法**：只看括号里的说明，能判断它描述的是目标文件的内容，而不是当前文件或另一个文件的内容吗？不能，就写错了。

---

### 标准七：模块覆盖必须和游戏实际系统对齐

总档里列出的所有主要系统，必须在 `02_模块拆解/` 或对应目录里有对应文件。没有对应文件的系统，等于该游戏的拆解有盲区。

**检验方法**：把总档的"主要系统"列表和实际文件列表对照，有没有系统存在于总档但没有对应拆解文件？有就是未完成，必须标注或补写。

## 快速自检清单

写完一篇专题后，用这个清单过一遍：

- [ ] 这篇文章里有没有至少一个具体的数值、条件或触发规则？
- [ ] 文章里有没有"特定 / 某些 / 一定程度 / 满足条件后"这类空位词？有就必须替换成具体内容。
- [ ] 开头的问题，能被反驳吗？
- [ ] 如果我做了功能分类，每个类别的识别标准写清楚了吗？
- [ ] 结论部分，放到另一个游戏的设计讨论里还成立吗？（太窄测试）
- [ ] 结论的 If 部分，包含了这类游戏特有的机制条件，而不是对任何游戏都成立的通用规律吗？（太宽测试）
- [ ] 这个模块"不负责什么"说清楚了吗？
- [ ] 交叉引用有方向和原因，不只是关联标注？括号里的描述指向的是目标文件，而不是其他文件？
- [ ] 总档的系统列表里，这个系统有没有对应的拆解文件覆盖？

九条全部 ✓，才算这篇专题合格。

## 深度不足的典型特征

以下情况说明一篇专题还没写够：

1. **全是骨架没有肉**：每一节只有一两句总结，读完还是不知道机制怎么运作
2. **用空位词回避具体机制**：写了"满足特定条件后触发"、"某些植物达到阈值"——这些和没写一样，必须替换成实际内容
3. **结论可以适用于任何游戏**："短循环高反馈"、"目标清晰"——这些对任何设计好的游戏都成立，没有信息量
4. **结论的 If 部分过于宽泛**：If-Then 格式是对的，但 If 条件里没有这类游戏的机制特征——换成任何其他游戏类型，结论仍然100%成立
5. **分类全是名称没有边界**：写了"路线件 / 底盘件 / 放大件"，但不写怎么判断一张牌属于哪类
6. **联动只写存在不写机制**："Joker 和经济系统联动"——联动在哪里？经济的哪个具体规则和 Joker 的哪个具体触发条件产生关系？
7. **检定节点没有具体案例**：说"Boss 专门检定偏科"，但没有列出任何一个具体 Boss 的机制，无法验证这个判断
8. **短板和检定缺失**：只写流派怎么成型，不写它在什么条件下会失败

---

## 这份标准本身的维护规则

这份文档是从实际拆解工作中提炼出来的，不是预设规范。

如果在拆解新游戏时发现某类问题反复出现，但这份文档没有覆盖，应当在此补充新的标准条目。

每条标准必须有对应的"不合格示例 → 合格示例"，否则不写入。
