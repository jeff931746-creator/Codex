## 入库流程

所有游戏拆解必须通过 `游戏机制拆解` Skill 完成，Skill 文件位于：
`/Users/mt/Documents/Codex/tools/codex-skills-repo/skills/游戏机制拆解/SKILL.md`

拆解完成后在此表格新增一行，才算正式入库。

## 目录检查

新增一个游戏时，至少要满足：

1. 有主目录
2. 有总档（`00_总档.md`），开头带 YAML frontmatter（字段见下）
3. 核心 4 专题齐全：`00_总档 / 02_模块拆解(≥2 文件) / 04_经济循环 / 90_结论提炼`
4. 按品类声明至少 2 个可选专题：`01_系统结构 / 03_流派设计 / 05_关卡节奏 / 06_商店与奖励 / 07_PVP与检定`
5. 总档正文里有一行 `本次拆解产出专题: [...]` 声明
6. 专题之间有明确边界
7. 最后有 If-Then 结论提炼（≥5 条，每条带「太宽测试」自检）

## Frontmatter 字段规范（总档开头）

```yaml
品类: [...]           # 多值数组。枚举：放置卡牌 / Roguelike构筑 / SLG / 塔防 / 模拟经营 / 修仙放置 / 射击肉鸽 / MMO / 卡牌对战 / 其他
平台: ...             # 单值。枚举：微信小游戏 / Steam / App手游 / 多平台
变现: ...             # 单值。枚举：IAP / 买断 / IAA / IAP+IAA
发行年份: 2024         # 4 位数字
核心循环: ...          # 一句话
研发商: ...
拆解日期: 2026-04-20
数据时点: 2024Q4      # 基于哪个时期的资料（版本/赛季/年份季度）
```

枚举外取值由脚本拒写。多视图段落（按品类/平台/变现/年份）由脚本自动聚合刷新。

## 专题命名建议

专题名尽量直接表达"要回答的问题"。

推荐形式：

- `牌种体系专题`
- `流派形成专题`
- `经济循环专题`
- `关卡与盲注专题`
- `商店修正专题`
- `构筑检定专题`

`02_模块拆解/` 下文件名**必须**以 `战斗_ / 养成_ / 构筑_ / 社交_` 之一开头，便于跨游戏检索（`ls 机制库/*/02_模块拆解/战斗_*`）。

不要把专题名写成抽象学科名，也不要写成泛泛的"机制分析"。

样本 ≥ 8 款后启用 `90_专题串联/`，不复制内容，只做链接 + 对比提炼。
