# 工作流链路规则

工作流全链路的层级架构、能力查找协议、阶段流程和目录约定。

---

## 一、层级架构

```
reference/                规则层（AI 不得擅改）— 固化后的标准与模板
  部门标准/              各部门执行标准，执行前必查
    立项/
    策划/
    开发/
  模板/                  项目结构模板
    立项预演/
    正式项目/

archive/                  沉淀层（经审批才入库）— 积累的知识与工具
  方法论/                已沉淀的分析框架和推导方法
  资料/                  外部收集的结构化数据
    机制库/
    竞品库/
    买量组合库/
    复盘库/
  经验/                  项目复盘和实战总结（经审批写入）
  skills/                可复用任务工作流
  tools/                 技术基础设施

workspace/                执行层 — 产出跟项目走，项目沿生命周期向下流动
  projects/
    方向探索/             发散+收敛阶段（话题级，方向未确定前）
      {话题名}/
        候选方向/         多个方向文档并列
        收敛结论.md       选定方向后填写，触发升格立项
    立项/                 已收敛到单一方向，走 6 阶段评审
      {项目名}/
    开发中/               立项通过后的开发阶段（部门制）
      {项目名}/
    上线/                 已上线项目
      {项目名}/
  playground/
  tmp/
```

**知识流向（单向升格，需审批）：**

```
workspace/ → archive/经验/ → archive/方法论/ → reference/
  (项目产出)   (审批后复盘)    (提炼为框架)     (固化为标准规则)
```

**两层模板的职责边界：**

| 模板 | 路径 | 适用阶段 | 组织方式 |
|---|---|---|---|
| 立项预演模板 | `reference/模板/立项预演/` | 立项评估阶段 | 按时间阶段（00-05） |
| 正式项目模板 | `reference/模板/正式项目/` | 立项通过后开发阶段 | 按部门+输入输出 |

## 二、能力查找协议（硬规则）

在执行任何需要专项能力的工作前（写策划文档、写代码、出美术需求等）：

```
1. 先查 reference/部门标准/{能力}/
2. 有标准 → 严格按标准执行，不得自行发挥或补充未经确认的内容
3. 无标准 → 立即停止，告知用户：
   "需要 [X] 能力的标准，当前 reference/部门标准/{能力}/ 下暂无，是否建立？"
4. 禁止在无标准的情况下自行创作任何能力性文档
```

**当前已有标准：**

| 能力域 | 路径 | 内容 |
|---|---|---|
| 立项流程 | `reference/部门标准/立项/立项流程机制/` | 六闸门评审、补证支线、纸面原型执行模板 |
| 系统策划 | `reference/部门标准/策划/系统策划/` | 资深系统策划 Prompt 母版、工作流与优化方案 |
| 开发 | `reference/部门标准/开发/` | 开发部门标准（持续扩充） |
| 美术 | `reference/部门标准/美术/` | 美术部门标准（预留，暂无内容） |

## 三、方向探索工作流（发散 → 收敛 → 升格）

适用于 `workspace/projects/方向探索/{话题名}/`，在立项之前进行。

| 阶段 | 操作 | 产物 |
|---|---|---|
| 发散 | 针对话题生成多个方向候选 | `候选方向/{方向名}.md`，每个方向一份文档，并列存放 |
| 收敛 | 评估候选方向，选定一个 | `收敛结论.md`：记录选定方向、选择理由、淘汰原因 |
| 升格 | 收敛完成后，进入正式立项 | 在 `立项/` 下新建 `{项目名}/`，复制模板，导入收敛结论 |

**流转规则：**
- 未完成收敛前，不得在 `立项/` 下建对应项目
- 淘汰的候选方向留在 `候选方向/` 原地，不删除，作为后续参考
- 话题长期无结论（搁置）→ 整个 `方向探索/{话题名}/` 移入 `archive/经验/`

---

## 四、立项阶段工作流（阶段制，6 阶段）

适用于 `workspace/projects/立项/{项目名}/`，阶段顺序如下：

| 阶段 | 目的 | 模板/标准路径 | 所需能力 | 完成标准 | 下一步 |
|---|---|---|---|---|---|
| 00-立项文档 | 定义核心玩法、竞品、长线假设 | `reference/部门标准/立项/立项流程机制/00-流程总览.md` | 策划 → `reference/部门标准/策划/` | 完成立项初审稿 | 进入 01-评审 |
| 01-评审 | 六闸门评审，决策通过/补证/否决 | `reference/部门标准/立项/立项流程机制/01-立项评审/` | 立项 → `reference/部门标准/立项/` | 评审结论明确 | 通过→建正式项目；补证→进 02；否决→归档 |
| 02-纸面原型 | 针对评审所需补证 | `reference/部门标准/立项/立项流程机制/02-纸面原型/` | 策划 → `reference/部门标准/策划/` | 补证材料完整，回到 01 复评 | 返回 01-评审 |
| 03-实现 | 开发实现 | `reference/部门标准/开发/` | 开发 → `reference/部门标准/开发/` | 功能可测试 | 进入 04-测试 |
| 04-测试 | 验证、Bug 修复 | 视情况查对应能力 | — | 测试通过 | 进入 05-发布 |
| 05-发布 | 上线、归档 | — | — | 完成发布记录 | 归档到正式项目目录 |

---

## 五、开发阶段项目结构（部门制）

立项通过后，将项目从 `workspace/projects/立项/` 移入 `workspace/projects/开发中/`，使用部门制目录，复制自 `reference/模板/正式项目/`。

```
{项目名}/
  立项/                  归档立项阶段文档（来自 workspace/projects/立项/）
  策划/
    输入/
      立项文档/          来自立项阶段的策划输入
      竞品参考/
      机制库引用/        从 archive/资料/机制库/ 引用
      用户调研/
    输出/
      系统设计/
      玩法规则/
      数值框架/
      GDD/               Game Design Document
  开发/
    输入/
      策划文档/          来自策划/输出/
      技术规范/
    输出/
      核心模块/
      配置表/
  美术/
    输入/
      策划需求/          来自策划/输出/
      风格参考/
    输出/
      概念图/
      UI设计/
      图标资产/
```

**部门间依赖关系：**
```
策划/输出/ → 开发/输入/策划文档/
策划/输出/ → 美术/输入/策划需求/
archive/资料/ → 策划/输入/（只读引用，不复制文件）
```

## 六、立项通过后的项目创建规则

```
项目生命周期（文件夹向下流动）：

workspace/projects/立项/{项目名}/
  ↓ 01-评审通过
workspace/projects/开发中/{项目名}/     ← 按 reference/模板/正式项目/ 建部门制目录
  ↓ 上线
workspace/projects/上线/{项目名}/
  ↓ 中止（任意阶段）
archive/经验/{项目名}/                  ← 复盘审批后沉淀

立项通过操作步骤：
1. git mv workspace/projects/立项/{项目名} workspace/projects/开发中/{项目名}
2. 在开发中/{项目名}/ 下建立部门制目录结构（复制自 reference/模板/正式项目/）
3. 将立项文档归入 开发中/{项目名}/立项/
```

---

## 七、资料引用规则

```
archive/资料/ 是只读引用层：
- 立项时参考 archive/方法论/立项白皮书/ 的推导方法
- 竞品分析参考 archive/资料/竞品库/
- 机制设计参考 archive/资料/机制库/
- 禁止直接修改 archive/ 下的资料和方法论文件
- 引用时注明来源路径，不复制整个文件到项目目录
```
