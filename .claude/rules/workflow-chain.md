# 工作流链路规则

工作流全链路的层级架构、能力查找协议、阶段流程和目录约定。

---

## 一、层级架构

```
research/                只读引用层 — 方法论、机制库、竞品库、复盘库
  立项白皮书/
  资料库/
    机制库/
    竞品库/
    买量组合库/

projects/                执行层
  00-共享能力/           能力标准层 — 各部门执行标准，执行前必查
    立项/                立项流程机制、评审模板、闸门表
    策划/                策划部门能力标准
    开发/                开发部门能力标准
    美术/                美术部门能力标准（预留）
  10-立项预演/           立项阶段池 — 立项评估中的项目（阶段制管理）
    _模板/               立项阶段项目模板（00-05 阶段结构）
    {项目名}/
  _正式项目模板/         开发阶段项目模板（部门制管理，详见第四节）
  {项目名}/              通过立项后建立的正式项目目录（部门制管理）
```

**两层模板的职责边界：**

| 模板 | 路径 | 适用阶段 | 组织方式 |
|---|---|---|---|
| 立项预演模板 | `10-立项预演/_模板/` | 立项评估阶段 | 按时间阶段（00-05） |
| 正式项目模板 | `_正式项目模板/` | 立项通过后开发阶段 | 按部门+输入输出 |

---

## 二、能力查找协议（硬规则）

在执行任何需要专项能力的工作前（写策划文档、写代码、出美术需求等）：

```
1. 先查 projects/00-共享能力/{能力}/
2. 有标准 → 严格按标准执行，不得自行发挥或补充未经确认的内容
3. 无标准 → 立即停止，告知用户：
   "需要 [X] 能力的标准，当前 00-共享能力/{能力}/ 下暂无，是否建立？"
4. 禁止在无标准的情况下自行创作任何能力性文档
```

**当前已有标准：**

| 能力域 | 路径 | 内容 |
|---|---|---|
| 立项流程 | `00-共享能力/立项/立项流程机制/` | 六闸门评审、补证支线、纸面原型执行模板 |
| 系统策划 | `00-共享能力/策划/系统策划/` | 资深系统策划 Prompt 母版、工作流与优化方案 |
| 开发 | `00-共享能力/开发/` | 开发部门标准（持续扩充） |
| 美术 | `00-共享能力/美术/` | 美术部门标准（预留，暂无内容） |

---

## 三、立项阶段工作流（阶段制，6 阶段）

适用于 `projects/10-立项预演/{项目名}/`，阶段顺序如下：

| 阶段 | 目的 | 模板/标准路径 | 所需能力 | 完成标准 | 下一步 |
|---|---|---|---|---|---|
| 00-立项文档 | 定义核心玩法、竞品、长线假设 | `00-共享能力/立项/立项流程机制/00-流程总览.md` | 策划 → `00-共享能力/策划/` | 完成立项初审稿 | 进入 01-评审 |
| 01-评审 | 六闸门评审，决策通过/补证/否决 | `00-共享能力/立项/立项流程机制/01-立项评审/` | 立项 → `00-共享能力/立项/` | 评审结论明确 | 通过→建正式项目；补证→进 02；否决→归档 |
| 02-纸面原型 | 针对评审所需补证 | `00-共享能力/立项/立项流程机制/02-纸面原型/` | 策划 → `00-共享能力/策划/` | 补证材料完整，回到 01 复评 | 返回 01-评审 |
| 03-实现 | 开发实现 | `00-共享能力/开发/` | 开发 → `00-共享能力/开发/` | 功能可测试 | 进入 04-测试 |
| 04-测试 | 验证、Bug 修复 | 视情况查对应能力 | — | 测试通过 | 进入 05-发布 |
| 05-发布 | 上线、归档 | — | — | 完成发布记录 | 归档到正式项目目录 |

## 四、开发阶段项目结构（部门制）

立项通过后，在 `projects/{项目名}/` 使用部门制目录，复制自 `_正式项目模板/`。

```
{项目名}/
  立项/                  归档立项阶段文档（从 10-立项预演 迁入）
  策划/
    输入/
      立项文档/          来自立项阶段的策划输入
      竞品参考/
      机制库引用/        从 research/资料库/机制库/ 引用
      用户调研/
    输出/
      系统设计/
      玩法规则/
      数值框架/
      GDD/               Game Design Document
  开发/
    输入/
      策划文档/          来自策划/输出/
      技术规范/
    输出/
      核心模块/
      配置表/
  美术/
    输入/
      策划需求/          来自策划/输出/
      风格参考/
    输出/
      概念图/
      UI设计/
      图标资产/
```

**部门间依赖关系：**
```
策划/输出/ → 开发/输入/策划文档/
策划/输出/ → 美术/输入/策划需求/
research/资料库/ → 策划/输入/（只读引用，不复制文件）
```

## 五、立项通过后的项目创建规则

```
立项通过（01-评审 结论为"通过"）后：
1. 在 projects/ 下创建 {项目名}/ 目录
2. 复制 _正式项目模板/ 的目录结构到新项目（保留 .gitkeep 占位）
3. 将立项阶段文档从 10-立项预演/{项目名}/ 迁入 {项目名}/立项/
4. 后续工作在新项目目录下按部门推进
5. 10-立项预演/{项目名}/ 归档或保留引用链接
```

---

## 六、research 引用规则

```
research/ 是只读引用层：
- 立项时参考 research/立项白皮书/ 的方法论
- 竞品分析参考 research/资料库/竞品库/
- 机制设计参考 research/资料库/机制库/
- 禁止在 research/ 下产出执行文件或项目文档
- 引用时注明来源路径，不复制整个文件到项目目录
```
